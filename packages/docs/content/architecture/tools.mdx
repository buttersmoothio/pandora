# Tools

## Capability-Based Permissions

Tools follow an Android-style permission model: every tool starts with zero authority and must declare exactly what it needs. There is no "admin" role or blanket access — a tool that needs network access declares which hostnames, a tool that needs time declares `time: true`.

This design means a tool can never silently escalate its capabilities. The permission declaration is part of the tool's manifest, visible to both the runtime and the UI before execution begins.

## Execution Tiers

Sandbox mode is not a per-tool option — it's determined by the tool's origin:

**Standard library tools** (`@pandora/tools-*` packages) always run in **host mode**. They execute directly in the Node.js process with full runtime access. Permissions are declared for auditing and UI display but are not enforced at runtime. These tools ship with the framework and are audited at development time.

**LLM-generated tools** (not yet implemented) will run in **compartment mode** — an SES Compartment with a blank global scope where only declared permissions are injected as endowments. Input and output cross the boundary via JSON serialization, stripping prototypes. This tier is designed for user-created tools where the host must not be trusted.

## Standard Library Packages

Built-in tools live in separate `@pandora/tools-*` packages rather than inside core. This keeps core's dependency surface small and lets each tool package be tested, versioned, and tree-shaken independently — the same pattern used by `@pandora/storage-*` and `@pandora/channel-*`.

Tools are explicit opt-in: only enabled tools are loaded. The default configuration enables the standard set so the experience is seamless out of the box, but operators have full control over which tools are active.

## Provider-Defined Tools

Not all tools are Pandora-defined. Some models have native tool support — OpenAI has `webSearch()`, Google has `googleSearch()`, Perplexity has built-in search. These are provider-defined tools from the Vercel AI SDK that don't go through `ToolExport` and have no Pandora manifest.

To accommodate this, `ToolRecord` is typed as Mastra's `ToolsInput` — a union that accepts Mastra tools, Vercel AI SDK tools, and provider-defined tools. This means a tool plugin can return any tool type the AI SDK understands, not just Pandora tools.

The `getTools` hook on `ToolPlugin` is the entry point for dynamic tools. It receives the operator's model string and returns the appropriate tools at load time. This is how `@pandora/tools-websearch` works: it checks the model, tries native search, then falls back to third-party search APIs.

Static tools (from `ToolExport` objects) and dynamic tools (from `getTools`) are complementary. Static tools have manifests, go through per-tool enable/disable, and show in the UI. Dynamic tools are added unconditionally when the plugin is active — they're opaque to the manifest system.

## Manifest Registry

Every tool has a manifest that keeps Pandora-specific metadata (permissions, sandbox mode, annotations) separate from the underlying Mastra tool object. This lets the UI display permission badges, sandbox mode, and approval prompts without coupling to the tool execution layer. Provider-defined tools bypass the manifest registry entirely — they have no Pandora metadata and are invisible to the `/api/tools` endpoint.
