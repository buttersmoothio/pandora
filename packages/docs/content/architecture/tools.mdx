# Tools

## Capability-Based Permissions

Tools follow an Android-style permission model: every tool starts with zero authority and must declare exactly what it needs. There is no "admin" role or blanket access — a tool that needs network access declares which hostnames, a tool that needs time declares `time: true`.

This design means a tool can never silently escalate its capabilities. The permission declaration is part of the tool's manifest, visible to both the runtime and the UI before execution begins.

## Execution Tiers

Sandbox mode is not a per-tool option — it's determined by the tool's origin:

**Standard library tools** (e.g. `@pandorakit/datetime`, `@pandorakit/tavily-search`) always run in **host mode**. They execute directly in the Node.js process with full runtime access. Permissions are declared for auditing and UI display but are not enforced at runtime. These tools ship with the framework and are audited at development time.

**LLM-generated tools** (not yet implemented) will run in **compartment mode** — an SES Compartment with a blank global scope where only declared permissions are injected as endowments. Input and output cross the boundary via JSON serialization, stripping prototypes. This tier is designed for user-created tools where the host must not be trusted.

## Standard Library Packages

Built-in tools live in separate packages (e.g. `@pandorakit/datetime`, `@pandorakit/tavily-search`) rather than inside core. This keeps core's dependency surface small and lets each tool package be tested, versioned, and tree-shaken independently — the same pattern used by `@pandorakit/storage-*` and `@pandorakit/telegram`.

Tools are explicit opt-in: only enabled tools are loaded. The default configuration enables the standard set so the experience is seamless out of the box, but operators have full control over which tools are active.

## Dynamic Tool Resolution

The `resolveTools` hook on `ToolPlugin` enables dynamic tool resolution. Unlike static `tools` (which are registered at startup), `resolveTools` runs at load time and returns `ToolExport` objects based on environment variables and config. This is how the search plugins (e.g. `@pandorakit/tavily-search`, `@pandorakit/brave-search`) work: each checks whether its API key is available and returns the appropriate backend as a `ToolExport`.

Resolved tools go through the same `bindToolExport()` + `registerManifest()` pipeline as static tools — they get manifests, timeouts, Zod schema wrapping, and appear in the UI. This means all tools, whether static or dynamically resolved, are first-class citizens with full manifest support.

## Model-Native Tools

Some models have built-in tool support — OpenAI has `webSearch()`, Google has `googleSearch()`, Anthropic has `webSearch_20250305()`. These are provider-defined tools that don't go through `ToolExport` and have no Pandora manifest.

Model-native tools are resolved in core via `resolveModelTools()`, not in plugins. Agents declare which model capabilities they need via `modelTools` in their manifest (e.g. `"modelTools": ["search"]`). Core maps these capability keys to the appropriate provider SDK tools based on the configured model.

## Manifest Registry

Every tool has a manifest that keeps Pandora-specific metadata (permissions, sandbox mode, annotations) separate from the underlying Mastra tool object. This lets the UI display permission badges, sandbox mode, and approval prompts without coupling to the tool execution layer.
