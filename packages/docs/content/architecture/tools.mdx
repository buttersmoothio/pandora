# Tools

## Capability-Based Permissions

Tools follow an Android-style permission model: every tool starts with zero authority and must declare exactly what it needs. There is no "admin" role or blanket access — a tool that needs network access declares which hostnames, a tool that needs time declares `time: true`.

This design means a tool can never silently escalate its capabilities. The permission declaration is part of the tool's manifest, visible to both the runtime and the UI before execution begins.

## Sandbox Modes

Tools execute in one of two modes:

**Compartment** (default) — code runs inside an SES Compartment with a blank global scope. Only endowed capabilities are accessible. Input and output cross the boundary via JSON serialization, stripping prototypes. This is the right choice for generated or external tool code where the host must not be trusted.

**Host** — the tool function runs directly in the Node.js process. Permissions are declared for auditing but not enforced at runtime. This is required for built-in tools that need closures, imports, or other host-process features that can't be serialized into a compartment.

The trade-off is safety vs. power. Compartment mode provides strong isolation guarantees; host mode provides full runtime access. Built-in tools use host mode because they ship with the framework and are audited at development time.

## Standard Library Packages

Built-in tools live in separate `@pandora/tools-*` packages rather than inside core. This keeps core's dependency surface small and lets each tool package be tested, versioned, and tree-shaken independently — the same pattern used by `@pandora/storage-*`.

Core discovers standard library packages at startup and dynamically imports each one. Every package exports a factory that receives environment variables and returns a tool record. Core merges these records and filters out any tool the user has not enabled in config.

The loading semantics are explicit opt-in: a tool is only available when enabled in config. The default configuration enables the standard set, so the experience is seamless out of the box, but operators have full control over which tools are active.

## Manifest Registry

Every tool created via `defineTool()` has a manifest registered in a global registry keyed by tool ID. This keeps Pandora-specific metadata (permissions, sandbox mode, annotations) separate from the Mastra Tool object while allowing lookup by ID without a tool instance.

The registry allows the UI to display descriptions, permission badges, sandbox mode, and approval prompts, and allows the runtime to select the correct execution mode. Tools created directly via Mastra's `createTool()` have no manifest — only Pandora-defined tools participate in the manifest system.
