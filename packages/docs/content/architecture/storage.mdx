# Storage Architecture

This page covers the internal design of Pandora's storage system.

## Two-Layer Design

Every storage provider produces a `StorageResult` with two independent layers:

```
StorageResult
в”ңв”Җв”Җ mastra: MastraCompositeStore   вҶҗ AI memory, threads, workflows
в””в”Җв”Җ config: ConfigStore<Config>    вҶҗ Agent configuration
```

**Mastra storage** handles AI-specific concerns: conversation threads, message history, workflow state, and vector embeddings. Each provider wraps a Mastra adapter package (`@mastra/libsql`, `@mastra/pg`, etc.).

**ConfigStore** handles Pandora-specific configuration: agent identity, model settings, channel configs, security, and tools. This is a simple key-value interface that each provider implements.

The two layers share the same database connection but manage separate tables/collections.

## Convention-Based Provider Discovery

Pandora discovers providers at runtime via dynamic `import()`:

```ts
// In packages/pandora/src/storage/index.ts
const provider = env.STORAGE_PROVIDER ?? 'libsql'
const mod = await import(`@pandora/storage-${provider}`)
const result = await mod.createStorage(env, bindings)
```

This approach means:
- No hardcoded provider list in the core package
- Third-party providers work automatically if they follow the `@pandora/storage-{name}` naming convention
- Missing providers produce clear installation instructions in the error message

After import, Pandora calls `mastra.init()` and `config.init()` (if defined) to set up schemas.

## Caching Strategy

в”Ӯ   Server Mode    в”Ӯ     в”Ӯ   Serverless Mode     в”Ӯ

In `getStorage()`, a module-level `_cached` variable stores the `StorageResult`. Server runtimes reuse this across requests. Serverless runtimes (detected via `getRuntimeKey()` returning `workerd`, `edge-light`, or `fastly`) always create fresh instances.

### Config Caching

Config adds a second cache layer:

1. **Storage cache** вҖ” the `StorageResult` itself (described above)
2. **Config cache** вҖ” the resolved `Config` object is cached in `_configCache` after first `getConfig()` call

In server mode, the config cache is updated on `PATCH` and cleared on `DELETE`. In serverless mode, neither cache is used.

## SQL Dialect Abstraction

`SQLConfigStore` supports three SQL dialects through a single class:

| Aspect | SQLite | PostgreSQL | MSSQL |
|---|---|---|---|
| JSON column type | `TEXT` | `JSONB` | `NVARCHAR(MAX)` |
| Param placeholders | `?` | `$1, $2` | `@p1, @p2` |
| Upsert | `ON CONFLICT DO UPDATE` | `ON CONFLICT DO UPDATE` | `MERGE` |
| Timestamp | `datetime('now')` | `NOW()` | `GETUTCDATE()` |
| Table existence check | `CREATE TABLE IF NOT EXISTS` | `CREATE TABLE IF NOT EXISTS` | `IF NOT EXISTS (SELECT * FROM sysobjects ...)` |

The dialect is set at construction time and determines which SQL templates are used. The class itself has zero driver-specific imports вҖ” callers provide an `execute(sql, params)` function wrapping their driver.

## Config Resolution Pipeline

When `getConfig()` is called, configuration is assembled in three stages:

```
в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ     в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ     в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
в”Ӯ  1. Defaults в”Ӯв”Җв”Җв”Җв”Җв–¶в”Ӯ  2. Storage   в”Ӯв”Җв”Җв”Җв”Җв–¶в”Ӯ 3. Env Vars  в”Ӯ
в”Ӯ  (schema)    в”Ӯ     в”Ӯ  (persisted)  в”Ӯ     в”Ӯ (overrides)  в”Ӯ
в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ     в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ     в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
                                                  в”Ӯ
                                                  в–ј
                                          в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
                                          в”Ӯ Zod validate  в”Ӯ
                                          в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
```

1. **Defaults** вҖ” `ConfigSchema.parse({})` produces a complete config with all default values
2. **Storage** вҖ” `configStore.get()` returns previously saved config, deep-merged onto defaults
3. **Env vars** вҖ” `loadFromEnv()` maps specific env vars to config paths (e.g., `PANDORA_NAME` вҶ’ `identity.name`)
4. **Validation** вҖ” The final merged config is validated through `ConfigSchema.parse()` to ensure type safety

Deep merging is recursive: nested objects are merged key-by-key, while arrays and primitives are replaced.

## Schema Validation

All configuration passes through Zod validation:

- `ConfigSchema.parse()` is called on every `getConfig()` вҖ” if any value is invalid, the request fails with a clear error
- `updateConfig()` validates after merging the patch, before persisting вҖ” invalid patches are rejected
- Default values are defined directly in the Zod schema using `.default()`, ensuring `ConfigSchema.parse({})` always produces a complete, valid config
