# Storage

Every storage provider produces three independent layers that share a database connection:

```
StorageResult
├── mastra   ← AI memory, threads, workflows (Mastra adapter)
├── config   ← Agent configuration (ConfigStore)
└── auth     ← Credentials, sessions, refresh tokens (AuthStore)
```

Providers are discovered at runtime via convention — `STORAGE_PROVIDER=postgres` imports `@pandora/storage-postgres`.

SQL-backed providers use dialect-aware helper classes (`SQLConfigStore`, `SQLAuthStore`) that handle SQLite, PostgreSQL, and MSSQL differences through a caller-provided `execute(sql, params)` function, keeping the classes free of driver-specific imports. KV-backed providers use equivalent helpers (`RedisConfigStore`, `RedisAuthStore`).

In server mode, all three layers are cached for the process lifetime. In serverless mode, fresh instances are created per request.
