# Authentication

Pandora uses a single-owner model — one password, no user accounts. This fits its role as a personal AI assistant rather than a multi-tenant service.

## Token Strategy

Authentication uses short-lived access tokens (15 min) paired with long-lived refresh tokens (7 days). On refresh, the old token is invalidated and a new pair is issued. If a used refresh token is presented again, all sessions are invalidated — this detects token theft without requiring token binding.

## Cryptography

All crypto uses the Web Crypto API for cross-runtime compatibility. Passwords are hashed with PBKDF2, session tokens are random bytes stored as hashes. All comparisons are timing-safe to prevent side-channel attacks.

## Auth Store

Auth persistence is decoupled from auth logic through the `AuthStore` interface, with SQL and KV implementations following the same pattern as storage providers.

## Request Pipeline

```
Request to /api/*
      │
      ▼
Rate limiter (auth endpoints only, in-memory per-IP)
      │
      ▼
Auth middleware
      │
      ├── Public path? → pass through
      ├── No credential set? → 403 setup_required
      ├── No/invalid token? → 401 unauthorized
      └── Valid token → continue
```
