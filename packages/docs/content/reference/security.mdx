# Security

Pandora is designed as a **single-owner personal assistant**. One person owns everything — all channels, conversations, and access.

## Core assumptions

1. **You are the only user** — No user accounts. One owner.
2. **Token = full access** — Anyone with your token can do everything you can.
3. **Cross-channel visibility is intentional** — The web UI shows Telegram conversations because they're yours.
4. **Trusted environment** — Pandora assumes it runs on your machine or private server.

## Authentication

### Telegram

Uses `ownerId` to restrict access:

```jsonc
"telegram": {
  "token": "bot-token",
  "ownerId": "your-telegram-user-id"
}
```

Only messages from `ownerId` are processed. Others are ignored.

### Web

Uses a shared secret token:

```jsonc
"web": {
  "port": 3000,
  "token": "your-secret-token"
}
```

Anyone with this token has full access — there are no user accounts.

## What "full access" means

With the web token, someone can:

- Send messages as you
- Read all conversation history
- Watch any conversation stream
- Clear or delete conversations
- Trigger any tool the AI has access to

This is by design. Pandora is your personal assistant, not a multi-tenant platform.

## Secrets management

`config.jsonc` contains your secrets:

- **Gitignored** — Won't be committed
- **File permissions** — Restrict with `chmod 600 config.jsonc`
- **Production** — Consider environment variables or secrets manager

## Deployment

### Local (default)

- Runs on `localhost:3000` (backend), `localhost:3001` (web UI)
- Only accessible from your machine
- Safe with default settings

### Private server

If running on a VPS:

1. Use a firewall — Only expose needed ports
2. Use HTTPS — Put a reverse proxy in front (nginx, Caddy)
3. Use strong tokens — Generate with `openssl rand -hex 32`

### Public internet

Pandora is **not designed for public deployment**. If you must:

1. Use HTTPS via reverse proxy
2. Add rate limiting at proxy level
3. Consider HTTP basic auth as second layer
4. Monitor logs for suspicious activity

## Not supported

Pandora does not provide:

- User accounts or registration
- Per-user permissions or roles
- Multi-tenant isolation
- OAuth or SSO
- Session management or token expiration

These would add complexity that conflicts with the "simple, hackable" philosophy.

## Threat model

### Protected against

| Threat | Mitigation |
|--------|------------|
| Unauthorized Telegram access | `ownerId` check rejects non-owners |
| Unauthorized web access | Token required for all requests |
| Config in git | `.gitignore` excludes `config.jsonc` |

### Your responsibility

| Threat | Mitigation |
|--------|------------|
| Token theft | Keep tokens secret, use HTTPS |
| Server compromise | Secure your server, restrict file permissions |
| Network sniffing | Use HTTPS in production |
| Malicious tools | Only install tools you trust |
