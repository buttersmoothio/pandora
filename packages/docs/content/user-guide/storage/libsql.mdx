# SQLite

SQLite is the default storage provider. It works out of the box with zero configuration, using a local database file.

## Installation

The SQLite provider is included as a default dependency — no extra install needed. If you're starting from scratch:

```bash
bun add @pandora/storage-libsql
```

## Configuration

| Variable | Required | Default | Description |
|---|---|---|---|
| `STORAGE_PROVIDER` | No | `libsql` | Set to `libsql` or omit entirely |
| `DATABASE_URL` | No | `file:data/pandora.db` | SQLite / LibSQL connection URL |
| `DATABASE_AUTH_TOKEN` | No | — | Auth token for remote Turso databases |

### Local File Mode

By default, Pandora creates a SQLite database at `data/pandora.db` relative to your project root. The directory is created automatically if it doesn't exist.

```bash
# No env vars needed — just start the server
bun run dev
```

You can customize the file path:

```bash
DATABASE_URL=file:/path/to/my.db
```

### Remote Mode (Turso)

To use a hosted [Turso](https://turso.tech) database:

```bash
DATABASE_URL=libsql://your-db-name-org.turso.io
DATABASE_AUTH_TOKEN=your-auth-token
```

## How It Works

The SQLite provider creates two storage layers:

1. **AI storage** — conversation threads, message history, and workflow state
2. **Config storage** — agent configuration in a `pandora_config` table (via `SQLConfigStore` with `sqlite` dialect)

Both share the same database connection. Schema tables are created automatically on first use.

## When to Use

- **Local development** — zero setup, file-based storage
- **Edge deployments** — embedded database with no external dependencies
- **Turso** — hosted SQLite with global replication
