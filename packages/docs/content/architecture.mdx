# Architecture

This page explains Pandora's internals. You don't need to read this to use Pandora — it's for those who want to understand how it works under the hood.

## Overview

```
┌─────────────────────────────────────────────────────────┐
│                     Your device                          │
│                 (Telegram, Browser)                      │
└─────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│                       Channel                            │
│             Receives messages, sends responses           │
└─────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│                       Gateway                            │
│        Routes messages, manages history, calls AI        │
└─────────────────────────────────────────────────────────┘
              │           │           │
              ▼           ▼           ▼
┌──────────────────┐  ┌──────────┐  ┌────────────────────┐
│      Store       │  │  Memory  │  │       Agent        │
│ Conversations    │  │ Episodic │  │ Operator + Agents  │
│                  │  │ Semantic │  │                    │
└──────────────────┘  └──────────┘  └────────────────────┘
```

## Components

### Channel

Connects to a messaging platform (Telegram, Web, Discord). Responsibilities:

- Listen for incoming messages
- Convert platform messages to Pandora's `Message` format
- Send responses back to users
- Handle platform-specific formatting

### Gateway

The central hub. Coordinates everything:

1. Receives message from channel
2. Saves message to store
3. Loads conversation history
4. Sends history to agent
5. Gets response from agent
6. Saves response to store
7. Returns response to channel

The Gateway also provides a pub/sub system for streaming events across channels.

### Store

Persists conversation history. Implementations:

- **SQLite** — File-based, persistent
- **Memory** — In-memory, for testing

### Memory

Provides long-term recall across conversations. Two types:

- **Episodic** — Automatic logging of interactions (gateway-managed)
- **Semantic** — Agent-controlled facts, preferences, knowledge

Uses vector embeddings for semantic search. Episodes are linked to conversations and cleaned up with them.

### Agent

The AI brain. Contains:

- **Operator** — Main AI that handles conversations
- **Sub-agents** — Specialists the operator delegates to

## Message flow

What happens when you send "Hello" to Telegram:

```
1. Telegram servers → Pandora webhook/polling
2. TelegramChannel.onMessage()
   - Extract text, user ID, chat ID
   - Build Message object
   - Call gateway.handleMessage(message, capabilities)
3. Gateway.handleMessage()
   - Save user message to store
   - Load conversation history
   - Auto-recall: Search memory for relevant context
   - Call agent.chat(history, capabilities, memoryContext)
4. Agent.chat()
   - Build system prompt with capabilities and tools
   - Send to AI model
   - Decide: respond directly or delegate
   - If delegating: call sub-agent, get result
   - Return final response
5. Gateway (continued)
   - Save AI response to store
   - Auto-episode: Store interaction in memory
   - Return response text
6. TelegramChannel (continued)
   - Format for Telegram (HTML, chunking)
   - Send via Telegram API
7. You see the response
```

## Delegation model

The operator can delegate to specialists:

```
User: "Write a Python function to sort a list"
         │
         ▼
┌─────────────────────────────────────────────────────────┐
│                       Operator                           │
│                                                          │
│  "This is a coding question. Delegating to coder."      │
│  → Calls the "coder" tool                               │
└─────────────────────────────────────────────────────────┘
         │
         ▼
┌─────────────────────────────────────────────────────────┐
│                     Coder Agent                          │
│                                                          │
│  Specialized instructions for programming               │
│  Returns detailed code response                          │
└─────────────────────────────────────────────────────────┘
         │
         ▼
┌─────────────────────────────────────────────────────────┐
│                       Operator                           │
│                                                          │
│  Receives coder's response, sends to user               │
└─────────────────────────────────────────────────────────┘
```

Each sub-agent appears as a tool to the operator. When called, the tool spins up that agent with the conversation context.

## Streaming events

The Gateway emits events during message processing:

| Event | Description |
|-------|-------------|
| `user-message` | User sent a message |
| `delta` | Text chunk from AI |
| `tool-call` | AI calling a tool |
| `tool-result` | Tool returned result |
| `source` | Citation URL |
| `reasoning-delta` | Chain-of-thought text |
| `step-finish` | Agent step completed |
| `done` | Response complete |

Clients subscribe via `gateway.subscribe(conversationId, listener)`.

## Cross-channel streaming

The web UI can watch any conversation from any channel:

1. Web UI sends `watch` message with conversation ID
2. Gateway registers the subscription
3. When that conversation gets activity (even from Telegram), events flow to web UI
4. Web UI updates in real-time

This enables:
- Viewing Telegram chats in browser
- Seeing responses stream across channels
- Unified conversation management

## Auto-discovery

Pandora finds extensions at startup:

```
1. Scan packages/pandora/src/tools/*.ts → Register tools
2. Scan packages/pandora/src/subagents/*.ts → Register agents
3. Scan packages/pandora/src/channels/*/index.ts → Register channels
4. Scan packages/pandora/src/store/*.ts → Register storage
5. Load config.jsonc
6. Create enabled components from config
7. Start channels
```

Files starting with `_` are skipped.

## Package structure

```
packages/
├── core/           @pandora/core (framework)
│   └── src/
│       ├── agent.ts      # Agent runtime
│       ├── gateway.ts    # Message routing hub
│       ├── config.ts     # Config loading/validation
│       ├── loader.ts     # Auto-discovery
│       └── registries/   # Extension registries
│
└── pandora/        @pandora/app (your extensions)
    └── src/
        ├── index.ts      # Entry point
        ├── tools/        # Tool implementations
        ├── subagents/    # Sub-agent definitions
        ├── channels/     # Channel implementations
        └── store/        # Storage backends
```

The separation means you can customize everything in `@pandora/app` without touching core.

## AI models

Pandora uses the [Vercel AI SDK](https://sdk.vercel.ai) and Vercel AI Gateway:

- **Single API key** — Access 50+ models
- **Any provider** — OpenAI, Anthropic, Google, etc.
- **Consistent interface** — Same code for any model

Model IDs use `provider/model` format:
- `google/gemini-3-flash` (recommended for operator)
- `anthropic/claude-sonnet-4.5` (recommended for coder)
- `openai/gpt-5.2` (recommended for research)
- `perplexity/sonar-pro` (recommended for web search)
