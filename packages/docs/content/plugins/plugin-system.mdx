# Plugin System

Pandora discovers plugins automatically -- drop a package with a `pandora.manifest.json` into the `packages/` directory and it appears in the UI. No manual registration needed.

## What's in a Plugin

A plugin is a standalone package that provides one or more of:

- **Tools** -- abilities the agent can use (web search, file operations, API calls)
- **Agents** -- specialist sub-agents that can be delegated to (research, coding)
- **Channels** -- connections to external messaging platforms (Telegram, Discord)

A single plugin can provide any combination of these.

## The Manifest

Every plugin has a `pandora.manifest.json` at its package root:

```json
{
  "$schema": "https://pandorakit.com/schemas/manifest-v1.json",
  "manifestVersion": 1,
  "id": "@pandorakit/my-plugin",
  "name": "My Plugin",
  "description": "What this plugin does",
  "author": "You",
  "version": "0.1.0",
  "license": "MIT",
  "homepage": "https://example.com",
  "repository": "https://github.com/you/my-plugin",
  "pandora": ">=0.0.1",
  "provides": { ... },
  "envVars": [{ "name": "MY_API_KEY" }],
  "configFields": [{ "key": "setting", "label": "Setting", "type": "text" }]
}
```

| Field | Required | Description |
|---|---|---|
| `manifestVersion` | Yes | Must be `1` |
| `id` | Yes | Plugin identifier -- matches the npm package name. Must not contain colons. |
| `name` | Yes | Human-readable display name |
| `pandora` | Yes | Semver range for core compatibility (e.g. `">=0.0.1"`) |
| `provides` | Yes | What the plugin provides: `tools`, `agents`, and/or `channels` |
| `description` | No | Short description shown in the UI |
| `author` | No | Plugin author |
| `icon` | No | Icon identifier for the UI |
| `version` | No | Plugin version |
| `homepage` | No | Link to the plugin's homepage |
| `repository` | No | Link to the source repository |
| `license` | No | License identifier |
| `envVars` | No | Environment variables the plugin needs -- shown in the UI with configured/missing status |
| `configFields` | No | User-configurable settings -- rendered as form fields in the UI |

### `envVars`

Each entry has a `name` and an optional `required` flag (defaults to `true` if omitted). Plugins with missing required env vars are skipped at load time.

```json
"envVars": [
  { "name": "MY_API_KEY" },
  { "name": "MY_OPTIONAL_VAR", "required": false }
]
```

### `provides`

The `provides` object declares what capabilities the plugin offers. Each key (`tools`, `agents`, `channels`) takes either a single entry or an array of entries.

**Tools entry:**

```json
"provides": {
  "tools": {
    "entry": "./src/index.ts",
    "sandbox": "compartment",
    "permissions": {
      "network": ["api.example.com"],
      "env": ["MY_API_KEY"]
    },
    "requireApproval": false
  }
}
```

| Field | Description |
|---|---|
| `entry` | Relative path to the entry module |
| `sandbox` | `"compartment"` (default) or `"host"` |
| `permissions` | Capabilities to grant in compartment mode |
| `requireApproval` | Default approval requirement for all tools in this entry |

**Agents entry:**

```json
"provides": {
  "agents": [
    {
      "entry": "./src/research.ts",
      "useTools": ["@pandorakit/tavily-search:tavily_search"],
      "modelTools": ["search"]
    }
  ]
}
```

| Field | Description |
|---|---|
| `entry` | Relative path to the entry module |
| `useTools` | Namespaced tool IDs to inherit from the global tool registry |
| `modelTools` | Model-native tool keys (e.g. `"search"` for built-in web search) |

Agents are always loaded in host mode regardless of any `sandbox` setting.

**Channels entry:**

```json
"provides": {
  "channels": {
    "entry": "./src/index.ts"
  }
}
```

Channels are always loaded in host mode.

## Namespacing

Tools, agents, and channels are registered with namespaced IDs: `pluginId:entityId` (e.g. `@pandorakit/tavily-search:tavily_search`). This prevents collisions between plugins that might use similar names. Entity IDs must not contain colons.

## Sandbox Model

Tool plugins can run in two modes:

- **`compartment`** (default) -- sandboxed. All capabilities are off by default -- no `fetch`, no `Date`, no env access. Each capability is individually granted via the `permissions` declaration in the manifest. TypeScript is supported (type erasure, preserving line numbers).
- **`host`** -- full process access. Use for tools that need native dependencies (WebSocket connections, platform SDKs).

Agents and channels always run in host mode regardless of the `sandbox` field.

### Permissions

Compartment-sandboxed tools declare what they need:

| Permission | What it grants |
|---|---|
| `time` | Access to `Date`, `Date.now()`, and `Intl.DateTimeFormat` |
| `network` | `fetch` scoped to declared hostnames only |
| `env` | Access to declared environment variable keys only (snapshot at load time) |
| `fs` | Filesystem read access scoped to declared path prefixes only |
| `random` | Access to `Math.random()` (cryptographic randomness is never available) |

Network access is SSRF-protected -- requests to private and internal addresses are blocked.

All sandboxed tools also receive these safe globals: `console` (routed to Pandora's logger), `URL`, `URLSearchParams`, `TextEncoder`, `TextDecoder`, `setTimeout`, `clearTimeout`, `setInterval`, `clearInterval`, `atob`, `btoa`, `queueMicrotask`, `AbortController`, and `AbortSignal`.

## Config Fields

Plugins can declare typed config fields that render as form inputs in the UI:

| Type | Renders as |
|---|---|
| `text` | Text input |
| `number` | Number input |
| `password` | Password input (masked) |
| `enum` | Dropdown select with `options` |

Each field supports `required`, `placeholder`, `description`, and (for `enum`) an `options` array of `{ value, label }` pairs.

Plugins with required config fields that aren't filled in are skipped at load time.

## Export Contracts

Each capability type expects specific exports from its entry module:

- **Tools**: `export const tools: Tool[]` and optionally `export function resolveTools(ctx): Promise<ResolveToolsResult>` for dynamic tool registration
- **Agents**: `export const agent: Agent` (one agent per entry -- use an array in `provides.agents` for multiple agents)
- **Channels**: `export const factory: ChannelFactory` -- a function that receives `(env, config)` and returns a `Channel` or `null`
