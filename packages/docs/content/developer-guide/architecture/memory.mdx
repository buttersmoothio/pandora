# Memory

## Separate from Storage

Storage and vector serve different workloads: storage handles structured data (threads, config, auth), while vector handles high-dimensional embedding search. Keeping them independent lets you mix backends — for example, Postgres for storage and a dedicated vector database for embeddings — without coupling unrelated concerns.

The default vector store uses SQLite, reading `DATABASE_URL` — the same env var as the default storage backend. This keeps zero-config setups simple while still allowing full separation when needed.

## Semantic Recall

Semantic recall is opt-in. When disabled (the default), no vector store is initialized and no embedding overhead is incurred. When enabled via config, the agent's `Memory` instance receives the vector store and an embedder string.

The embedding model is resolved via Mastra's model router using a `provider/model` string format (e.g. `openai/text-embedding-3-small`). Mastra defaults are used for `topK` and `messageRange` rather than re-inventing tuning parameters — the goal is to expose a simple on/off toggle to the user while deferring to Mastra's sensible defaults for retrieval behavior.
