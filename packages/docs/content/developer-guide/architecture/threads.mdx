# Threads & Branching

Conversations are stored as threads. When a user edits a previous message, the system forks the thread into a new branch rather than mutating history — every response is preserved and reachable.

## Why Clone-Based Forking

Branches are implemented as Mastra memory thread clones. Each branch is a full, independent thread that can be loaded, continued, and deleted without affecting its siblings or parent. This avoids a custom tree data structure in the storage layer and works with any Mastra-compatible storage provider out of the box.

## Thread Families

Threads are either roots (new conversations) or branches (forks). Only root threads appear in listings. When a thread is forked, the clone carries a pointer to which message it diverges at. Siblings are discovered at read time by querying the parent's clones for matching fork points.

Each root thread is enriched with its full family of branches and which branch was most recently active. This lets sidebar UIs show one entry per conversation while linking to the active branch.

## Fork-Point Resolution

Fork points are tracked by message ID, not by array index. The `forkPointIndex` returned in the API is computed on read by filtering to user/assistant messages (matching what the UI displays) and finding the fork-point message's position. This keeps the stored metadata stable even if system messages are added or removed.
