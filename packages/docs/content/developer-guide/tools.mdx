# Custom Tools

Tools extend what your Pandora agent can do — fetch data, read files, call APIs, or anything else you can express as a function. Every tool declares its permissions upfront and runs in a sandboxed or host environment.

For the full type reference (all fields, interfaces, and options), see the [Tools Reference](/reference/tools).

## Building a Tool

<Steps>

### Define the tool

Use `defineTool()` to create a tool with a permission manifest. Here's a weather tool that fetches data from an external API:

```ts
import { z } from 'zod'
import { defineTool } from '@pandora/core/tools'

const weatherTool = defineTool({
  id: 'weather',
  description: 'Get current weather for a city',
  inputSchema: z.object({
    city: z.string().describe('City name'),
  }),
  permissions: {
    network: ['api.openweathermap.org'],
    env: ['OPENWEATHER_API_KEY'],
  },
  sandbox: 'host',
  annotations: {
    readOnlyHint: true,
    destructiveHint: false,
    idempotentHint: true,
  },
  execute: async (input) => {
    const key = process.env.OPENWEATHER_API_KEY
    const res = await fetch(
      `https://api.openweathermap.org/data/2.5/weather?q=${input.city}&appid=${key}`
    )
    return res.json()
  },
})
```

A few things to notice:

- **`permissions`** declares exactly what the tool needs. `network` lists allowed hostnames, `env` lists environment variables. A tool with `{}` gets only `console` access.
- **`sandbox: 'host'`** runs the function directly in Node.js. The default (`'compartment'`) runs it in an isolated SES Compartment — use that for untrusted or generated code.
- **`annotations`** are MCP-compatible hints that help the UI decide whether to auto-approve or prompt for confirmation.

### Package it

Pandora's built-in tools ship as separate `@pandora/tools-*` packages. Each package exports a `createTools` factory that returns a tool record:

```ts
// @pandora/tools-datetime/src/index.ts
import type { ToolPackageFactory } from '@pandora/core/tools'
import { currentTime } from './current-time'

export const createTools: ToolPackageFactory = (_env) => ({
  'current-time': currentTime,
})
```

The factory receives environment variables so tools can read API keys or other configuration at load time.

A tool package follows this structure:

<FileTree>
  <FileTree.Folder name="packages/tools-example" defaultOpen>
    <FileTree.File name="package.json" />
    <FileTree.Folder name="src" defaultOpen>
      <FileTree.File name="index.ts" />
      <FileTree.File name="my-tool.ts" />
      <FileTree.File name="my-tool.test.ts" />
    </FileTree.Folder>
    <FileTree.File name="tsconfig.json" />
  </FileTree.Folder>
</FileTree>

The `package.json` declares `@pandora/core` as a peer dependency:

```json
{
  "name": "@pandora/tools-example",
  "type": "module",
  "exports": { ".": "./src/index.ts" },
  "peerDependencies": {
    "@pandora/core": "workspace:*"
  },
  "dependencies": {
    "zod": "^4.3.6"
  }
}
```

### Enable it

Core loads standard library packages dynamically and filters tools based on the user's configuration — only tools with `config.tools[id].enabled === true` are loaded. The default configuration enables the standard set.

</Steps>

## Permissions

Every tool starts with zero authority. You declare what it needs in a `permissions` object:

```ts
defineTool({
  permissions: {
    time: true,                          // Date, Date.now(), Intl.DateTimeFormat
    network: ['api.example.com'],        // fetch scoped to these hostnames
    env: ['API_KEY'],                    // read-only access to these env vars
    fs: ['/data'],                       // read-only filesystem under these paths
    random: true,                        // Math.random() (not cryptographic)
  },
  // ...
})
```

An empty `{}` grants only `console` access (always available in compartment mode).

In **compartment** mode, permissions map directly to globals injected into the sandbox. In **host** mode, permissions are recorded in the manifest for auditing and UI display but are not runtime-enforced.

## Sandbox Modes

**Compartment** (default) — code runs inside an SES Compartment with an isolated global scope. Only endowed capabilities (derived from permissions) are available. Input and output are JSON-serialized to strip prototypes. Use this for generated or untrusted tool code.

**Host** — the function runs directly in Node.js. Use this when you need closures, imports, or other Node.js features.

```ts
// Compartment mode (default) — isolated execution
defineTool({
  sandbox: 'compartment', // or omit — this is the default
  // ...
})

// Host mode — full Node.js access
defineTool({
  sandbox: 'host',
  // ...
})
```

## Manifests

Every `defineTool()` call registers a manifest in a global registry keyed by tool ID. This lets the UI display descriptions, permission badges, and approval prompts, and lets the runtime select the correct execution mode.

```ts
import { getAllManifests, getManifest, getManifests } from '@pandora/core/tools'

const manifest = getManifest(weatherTool)
const manifests = getManifests(toolRecord)
const all = getAllManifests() // every registered tool
```

Only tools created via `defineTool()` have manifests. Raw Mastra tools created with `createTool()` return `undefined`.

## Compartment Execution

For advanced use cases, you can execute code strings directly in an SES Compartment:

```ts
import { executeInCompartment } from '@pandora/core/tools'

const result = await executeInCompartment({
  code: `async function(input) {
    const res = await fetch('https://api.example.com/data?q=' + input.query)
    return res.json()
  }`,
  input: { query: 'hello' },
  permissions: { network: ['api.example.com'] },
  envVars: process.env,
})
```

Output crosses the compartment boundary via JSON serialization. Plain data passes through unchanged, `undefined` becomes `null`, functions and symbols are stripped, and circular references throw.
