# Build a Tool

Tools are standalone npm packages that run in Node.js with full runtime access.

<Steps>

### Set up the package

A tool package follows this structure:

<FileTree>
  <FileTree.Folder name="pandora-tools-weather" defaultOpen>
    <FileTree.File name="package.json" />
    <FileTree.Folder name="src" defaultOpen>

      <FileTree.File name="weather.test.ts" />
    </FileTree.Folder>
    <FileTree.File name="tsconfig.json" />
  </FileTree.Folder>
</FileTree>

The `package.json` declares `@pandora/core` as a peer dependency:

```json
{
  "name": "pandora-tools-weather",
  "type": "module",
  "exports": { ".": "./src/index.ts" },
  "peerDependencies": {
    "@pandora/core": "*"
  },
  "dependencies": {
    "zod": "^4.3.6"
  }
}
```

### Define the tool

Use `defineTool()` to create a tool. Here's a weather tool that fetches data from an external API:

```ts
// src/weather.ts
import { z } from 'zod'
import { defineTool } from '@pandora/core/tools'

export const weatherTool = defineTool({
  id: 'weather',
  name: 'Weather',
  description: 'Get current weather for a city',
  inputSchema: z.object({
    city: z.string().describe('City name'),
  }),
  annotations: {
    readOnlyHint: true,
    destructiveHint: false,
    idempotentHint: true,
  },
  execute: async (input, { env, config }) => {
    const key = env.OPENWEATHER_API_KEY
    const units = config.units ?? 'metric'
    const res = await fetch(
      `https://api.openweathermap.org/data/2.5/weather?q=${input.city}&units=${units}&appid=${key}`
    )
    return res.json()
  },
})
```

- `defineTool()` returns a **tool definition** that you list in your plugin's `tools` array. The `execute` function receives `{ env, config }` as its second argument, giving tools clean access to environment variables and plugin configuration.
- **`annotations`** are MCP-compatible hints that help the UI decide whether to auto-approve or prompt for confirmation. A tool marked `readOnlyHint: true` and `destructiveHint: false` can be auto-approved.

### Export the plugin descriptor

Each package default-exports a `ToolPlugin` descriptor with metadata and a `tools` array listing the tool definitions:

```ts
// src/index.ts
import type { ToolPlugin } from '@pandora/core/tools'
import { weatherTool } from './weather'

export default {
  id: 'tools-weather',
  name: 'Weather',
  schemaVersion: 1,
  envVars: [{ name: 'OPENWEATHER_API_KEY' }],
  configFields: [
    {
      key: 'units',
      label: 'Units',
      type: 'enum',
      description: 'Temperature units',
      options: [
        { value: 'metric', label: 'Metric (Â°C)' },
        { value: 'imperial', label: 'Imperial (Â°F)' },
        { value: 'standard', label: 'Standard (K)' },
      ],
    },
  ],
  tools: [weatherTool],
} satisfies ToolPlugin
```

- **`name`** â€” human-readable display name for the UI.
- **`tools`** â€” array of tool definitions returned by `defineTool()`. Pandora binds `(env, config)` automatically when loading plugins â€” you just list them.
- **`envVars`** â€” environment variables the plugin needs, as `{ name, required? }` descriptors. Variables default to required; set `required: false` for optional ones. The settings UI shows whether they're configured.
- **`configFields`** â€” describes user-configurable settings beyond `enabled`. Pandora validates input and renders fields in the settings UI from these descriptors.
- If `configFields` are declared and the user hasn't configured them, the plugin is skipped.

### Register it

Install your package in the Pandora instance and register it in `plugins.ts`:

```ts
import weather from 'pandora-tools-weather'
import { registerToolPlugin } from './tools'

registerToolPlugin(weather)
```

### Enable it

Once registered, users control your plugin at two levels in **Settings â†’ Tools**:

- **Plugin level** â€” toggles the plugin on/off and stores settings from your `configFields`. A plugin with required `configFields` is skipped until the user configures them.
- **Tool level** â€” toggles individual tools within the plugin. Only enabled tools are loaded.

## Dynamic Tools

Some tools can't be defined statically â€” they depend on which model the user selected or which API keys are available. For these cases, `ToolPlugin` supports an async `getTools` hook that runs at load time and returns tools dynamically.

This is the mechanism behind `@pandora/tools-websearch`: it detects the operator's model and returns the appropriate provider tool (OpenAI's `webSearch()`, Google's `googleSearch()`, or a third-party search API like Tavily).

```ts
import type { ToolPlugin } from '@pandora/core/tools'
import { resolveSearchTools } from './resolve'

export default {
  id: 'tools-websearch',
  name: 'Web Search',
  schemaVersion: 1,
  configFields: [
    {
      key: 'searchBackend',
      label: 'Search Backend',
      type: 'enum',
      options: [
        { value: 'auto', label: 'Auto-detect' },
        { value: 'tavily', label: 'Tavily' },
        { value: 'exa', label: 'Exa' },
      ],
    },
  ],
  tools: [],
  async getTools({ model, pluginConfig, env }) {
    const preferred = pluginConfig?.searchBackend as string | undefined
    return (await resolveSearchTools({ model, preferred, env })) ?? {}
  },
} satisfies ToolPlugin
```

A plugin can have both static `tools` and a `getTools` hook. Static tools go through the normal per-tool enable/disable flow; dynamic tools are added unconditionally when the plugin is active.

The `getTools` hook receives a `GetToolsContext`:

```ts
interface GetToolsContext {
  model: string                          // e.g. 'openai/gpt-4o'
  pluginConfig: Record<string, unknown>  // validated plugin config
  env: Record<string, string | undefined>
}
```

Return a `ToolRecord` â€” a record of tool instances keyed by ID. This can include Mastra tools, Vercel AI SDK tools, or provider-defined tools (like `openai.tools.webSearch()`). Return `{}` if no tools are available.

## Reference

All types are exported from `@pandora/core/tools`:

```ts
import { defineTool } from '@pandora/core/tools'

import type {
  ConfigFieldDescriptor,
  EnvVarDescriptor,
  ToolExecuteContext,
  ToolAnnotations,
  ToolPlugin,
  ToolPluginConfig,
} from '@pandora/core/tools'
```

### `ToolExecuteContext`

The second argument passed to the tool's `execute` function:

```ts
interface ToolExecuteContext {
  env: Record<string, string | undefined>
  config: ToolPluginConfig
}
```

### `DefineToolOptions`

| Field | Type | Required | Description |
|---|---|---|---|
| `id` | `string` | Yes | Unique tool identifier |
| `name` | `string` | Yes | Human-readable display name |
| `description` | `string` | Yes | What the tool does (shown to the LLM) |
| `inputSchema` | `z.ZodType` | Yes | Zod schema for input validation |
| `outputSchema` | `z.ZodType` | No | Optional Zod schema for output validation |
| `execute` | `(input, context: ToolExecuteContext) => Promise` | Yes | The tool's handler function |
| `permissions` | `ToolPermissions` | No | Declared capabilities (time, network, env, fs, random) |
| `sandbox` | `'compartment' \| 'host'` | No | `'host'` runs with full Node.js access (default for tool plugins). `'compartment'` runs in a restricted sandbox. |
| `timeout` | `number` | No | Execution timeout in milliseconds. Defaults to `60000` (60 s) |
| `requireApproval` | `boolean` | No | Require user approval before execution |
| `annotations` | `ToolAnnotations` | No | MCP-compatible UI hints |

### `ToolAnnotations`

```ts
interface ToolAnnotations {
  title?: string              // Display title in UI
  readOnlyHint?: boolean      // Safe to run without confirmation
  destructiveHint?: boolean   // May perform irreversible operations
  idempotentHint?: boolean    // Same args = no additional effect
}
```

### `ToolPluginConfig`

The validated user configuration passed to tool definitions:

```ts
interface ToolPluginConfig {
  enabled: boolean
  [key: string]: unknown // custom fields from configFields
}
```

### `ToolPlugin`

The descriptor your package default-exports:

| Field | Type | Required | Description |
|---|---|---|---|
| `id` | `string` | Yes | Plugin identifier, e.g. `'tools-datetime'` |
| `name` | `string` | Yes | Human-readable display name, e.g. `'Date & Time'` |
| `schemaVersion` | `number` | Yes | Must be `1` |
| `envVars` | `EnvVarDescriptor[]` | No | Environment variable descriptors |
| `configFields` | `ConfigFieldDescriptor[]` | No | UI config field descriptors |
| `tools` | `ToolDefinition[]` | Yes | Tool definitions returned by `defineTool()` |
| `getTools` | `(ctx: GetToolsContext) => Promise<ToolRecord>` | No | Async hook for dynamic tool resolution. See [Dynamic Tools](#dynamic-tools). |

### `EnvVarDescriptor`

Describes an environment variable a plugin depends on:

| Field | Type | Required | Description |
|---|---|---|---|
| `name` | `string` | Yes | Environment variable name |
| `required` | `boolean` | No | Whether this variable is required. Defaults to `true` |
