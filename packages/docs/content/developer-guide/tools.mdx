# Custom Tools

Tools extend what your Pandora agent can do — fetch data, read files, call APIs, or anything else you can express as a function. Every tool declares its permissions upfront and runs in a sandboxed or host environment.

## Core Types

All tool types are exported from `@pandora/core/tools`:

```ts
import {
  defineTool,
  getManifest,
  getManifests,
  executeInCompartment,
} from '@pandora/core/tools'

import type {
  ToolPermissions,
  ToolManifest,
  ToolAnnotations,
  SandboxMode,
  ToolRecord,
  Endowments,
  CompartmentExecuteOptions,
} from '@pandora/core/tools'
```

## Defining a Tool

Use `defineTool()` to create a tool with a permission manifest:

```ts
import { z } from 'zod'
import { defineTool } from '@pandora/core/tools'

const weatherTool = defineTool({
  id: 'weather',
  description: 'Get current weather for a city',
  inputSchema: z.object({
    city: z.string().describe('City name'),
  }),
  permissions: {
    network: ['api.openweathermap.org'],
    env: ['OPENWEATHER_API_KEY'],
  },
  sandbox: 'host',
  annotations: {
    readOnlyHint: true,
    destructiveHint: false,
    idempotentHint: true,
  },
  execute: async (input) => {
    const key = process.env.OPENWEATHER_API_KEY
    const res = await fetch(
      `https://api.openweathermap.org/data/2.5/weather?q=${input.city}&appid=${key}`
    )
    return res.json()
  },
})
```

### `DefineToolOptions`

| Field | Type | Required | Description |
|---|---|---|---|
| `id` | `string` | Yes | Unique tool identifier |
| `description` | `string` | Yes | What the tool does (shown to the LLM) |
| `inputSchema` | `z.ZodType` | Yes | Zod schema for input validation |
| `outputSchema` | `z.ZodType` | No | Optional Zod schema for output validation |
| `execute` | `(input, context) => Promise` | Yes | The tool's handler function |
| `permissions` | `ToolPermissions` | Yes | Declared capabilities (see below) |
| `sandbox` | `SandboxMode` | No | `'compartment'` (default) or `'host'` |
| `requireApproval` | `boolean` | No | Require user approval before execution |
| `annotations` | `ToolAnnotations` | No | MCP-compatible UI hints |

## Permissions

Every tool declares a `ToolPermissions` object describing what capabilities it needs. Tools start with zero authority — only declared permissions are granted.

```ts
interface ToolPermissions {
  time?: boolean        // Date, Date.now(), Intl.DateTimeFormat
  network?: string[]    // fetch scoped to these hostnames
  env?: string[]        // Read-only access to these env vars
  fs?: string[]         // Read-only filesystem under these path prefixes
  random?: boolean      // Math.random() (not cryptographic)
}
```

| Permission | Grants | Example |
|---|---|---|
| `time: true` | `Date`, `Date.now()`, `Intl.DateTimeFormat` | Formatting timestamps |
| `network: ['api.example.com']` | `fetch` scoped to listed hostnames | API calls |
| `env: ['API_KEY']` | `env.get('API_KEY')` | Reading secrets |
| `fs: ['/data']` | `readFile()` under `/data` | Reading local files |
| `random: true` | `Math.random()` | Generating IDs |

An empty `{}` grants only `console` access (always available in compartment mode).

## Sandbox Modes

```ts
type SandboxMode = 'compartment' | 'host'
```

**Compartment** (default) — code runs inside an SES Compartment with an isolated global scope. Only endowed capabilities (derived from permissions) are available. Input and output are JSON-serialized to strip prototypes.

Use compartment mode for generated or untrusted tool code.

**Host** — the function runs directly in Node.js. Permissions are recorded in the manifest for auditing and UI display but are not runtime-enforced.

Use host mode when you need closures, imports, or other Node.js features.

```ts
// Compartment mode (default) — isolated execution
defineTool({
  id: 'my-tool',
  sandbox: 'compartment', // or omit — this is the default
  permissions: { network: ['api.example.com'] },
  // ...
})

// Host mode — full Node.js access
defineTool({
  id: 'my-host-tool',
  sandbox: 'host',
  permissions: { network: ['api.example.com'] },
  // ...
})
```

## Annotations

MCP-compatible hints that help the UI decide how to present the tool:

```ts
interface ToolAnnotations {
  title?: string              // Display title in UI
  readOnlyHint?: boolean      // Safe to run without confirmation
  destructiveHint?: boolean   // May perform irreversible operations
  idempotentHint?: boolean    // Same args = no additional effect
}
```

A tool marked `readOnlyHint: true` and `destructiveHint: false` can be auto-approved by the UI. A tool with `destructiveHint: true` should always prompt for confirmation.

## Manifests

Every `defineTool()` call registers a `ToolManifest` in an internal registry. Use `getManifest()` to inspect a tool's metadata:

```ts
import { getManifest, getManifests } from '@pandora/core/tools'

// Single tool
const manifest = getManifest(weatherTool)
// { id: 'weather', description: '...', permissions: {...}, sandbox: 'host', annotations: {...} }

// All tools in a record
const manifests = getManifests(toolRecord)
// { weather: {...}, 'current-time': {...} }
```

Only tools created via `defineTool()` have manifests. Raw Mastra tools created with `createTool()` return `undefined`.

## Compartment Execution

For advanced use cases, you can execute code strings directly in an SES Compartment:

```ts
import { executeInCompartment } from '@pandora/core/tools'

const result = await executeInCompartment({
  code: `async function(input) {
    const res = await fetch('https://api.example.com/data?q=' + input.query)
    return res.json()
  }`,
  input: { query: 'hello' },
  permissions: { network: ['api.example.com'] },
  envVars: process.env,
})
```

### `CompartmentExecuteOptions`

| Field | Type | Description |
|---|---|---|
| `code` | `string` | Function body — must evaluate to `async function(input) { ... }` |
| `input` | `unknown` | Input data passed to the function |
| `permissions` | `ToolPermissions` | Determines which endowments are injected |
| `envVars` | `Record<string, string \| undefined>` | Environment variables for scoped env access |

### Endowments

In compartment mode, permissions map to globals injected into the compartment:

```ts
interface Endowments {
  fetch?: (url: string, init?: RequestInit) => Promise<Response>
  env?: { get: (key: string) => string | undefined }
  readFile?: (path: string) => Promise<string>
  Date?: typeof Date
  Intl?: typeof Intl
  Math?: { random: () => number }
  console: Pick<Console, 'log' | 'warn' | 'error'>  // always present
}
```

All endowments are `harden()`-ed (frozen via SES) to prevent tampering.

### Output Serialization

Output crosses the compartment boundary via `JSON.stringify` → `JSON.parse`. This means:

- **Plain data** (objects, arrays, strings, numbers, booleans, null) passes through unchanged
- **`undefined`** becomes `null`
- **Functions and symbols** are silently stripped from objects
- **Circular references** throw a serialization error
- **Output exceeding 1 MB** throws a size limit error
