# Build a Tool

Tool plugins are standalone packages that provide tools Pandora can use. Each plugin is automatically discovered via its `pandora.manifest.json` â€” no manual registration needed.

<Steps>

### Set up the package

A tool package follows this structure:

<FileTree>
  <FileTree.Folder name="weather" defaultOpen>
    <FileTree.File name="package.json" />
    <FileTree.File name="pandora.manifest.json" />
    <FileTree.Folder name="src" defaultOpen>
      <FileTree.File name="index.ts" />
      <FileTree.File name="weather.ts" />
    </FileTree.Folder>
    <FileTree.File name="tsconfig.json" />
  </FileTree.Folder>
</FileTree>

The `package.json` declares `@pandorakit/sdk` as a dev dependency (only type-only imports are used at runtime):

```json
{
  "name": "@pandorakit/weather",
  "type": "module",
  "exports": { ".": "./src/index.ts" },
  "devDependencies": {
    "@pandorakit/sdk": "*"
  }
}
```

### Write the manifest

The `pandora.manifest.json` declares your plugin's identity, capabilities, and requirements:

```json
{
  "manifestVersion": 1,
  "id": "@pandorakit/weather",
  "name": "Weather",
  "description": "Get current weather for any city",
  "author": "You",
  "version": "0.1.0",
  "license": "MIT",
  "pandora": ">=0.0.1",
  "provides": {
    "tools": {
      "entry": "./src/index.ts",
      "sandbox": "compartment",
      "permissions": {
        "network": ["api.openweathermap.org"],
        "env": ["OPENWEATHER_API_KEY"]
      }
    }
  },
  "envVars": [{ "name": "OPENWEATHER_API_KEY" }],
  "configFields": [
    {
      "key": "units",
      "label": "Units",
      "type": "enum",
      "description": "Temperature units",
      "options": [
        { "value": "metric", "label": "Metric" },
        { "value": "imperial", "label": "Imperial" }
      ]
    }
  ]
}
```

Key manifest fields:

| Field | Description |
|---|---|
| `manifestVersion` | Must be `1` |
| `id` | Plugin identifier â€” matches the npm package name |
| `pandora` | Semver range for core compatibility (e.g. `">=0.0.1"`) |
| `provides.tools.entry` | Path to the entry point that exports the tools |
| `provides.tools.sandbox` | `"compartment"` (default, sandboxed) or `"host"` (full access) |
| `provides.tools.permissions` | Capabilities the tools need: `time`, `network`, `env`, `fs`, `random` |
| `envVars` | Environment variables the plugin needs â€” shown in the UI with configured/missing status |
| `configFields` | User-configurable settings beyond `enabled` â€” rendered as form fields in the UI |

### Define the tool

Export a `Tool` object â€” a plain object with metadata, JSON Schema parameters, and an `execute` function:

```ts
// src/weather.ts
import type { Tool } from '@pandorakit/sdk/tools'

export const weatherTool: Tool = {
  id: 'weather',
  name: 'Weather',
  description: 'Get current weather for a city',
  parameters: {
    type: 'object',
    properties: {
      city: { type: 'string', description: 'City name' },
    },
    required: ['city'],
  },
  annotations: {
    readOnlyHint: true,
    destructiveHint: false,
    idempotentHint: true,
  },
  execute: async (input: { city: string }, { env }) => {
    const key = env.OPENWEATHER_API_KEY
    const res = await fetch(
      `https://api.openweathermap.org/data/2.5/weather?q=${input.city}&units=metric&appid=${key}`
    )
    return res.json()
  },
}
```

- `Tool` is a plain object â€” no framework dependencies at runtime. Only type-only imports from `@pandorakit/sdk` (erased by the TypeScript compiler).
- The `execute` function receives `{ env }` as its second argument, giving tools access to environment variables.
- **`annotations`** are MCP-compatible hints that help the UI decide whether to auto-approve or prompt for confirmation.

### Export the tools array

The entry point (declared in the manifest's `provides.tools.entry`) exports a named `tools` array:

```ts
// src/index.ts
import { weatherTool } from './weather'

export const tools = [weatherTool]
```

Tools are registered with namespaced IDs: `pluginId:toolId` (e.g. `@pandorakit/weather:weather`).

### Enable it

Pandora automatically discovers your plugin via its `pandora.manifest.json`. Users control it from the **Plugins** page:

- Toggle the plugin on/off
- Configure settings from your `configFields`
- Optionally enable **Require Approval** on individual tools

A plugin with required `configFields` is skipped until the user configures them. Plugins with only optional config fields load with defaults automatically.

</Steps>

## Dynamic Tool Resolution

Some tools can't be defined statically â€” they depend on which API keys are available or user configuration. For these cases, your entry point can export an async `resolveTools` function that runs at load time and returns `Tool` objects dynamically.

This is the mechanism behind the search plugins (e.g. `@pandorakit/tavily-search`, `@pandorakit/brave-search`): each checks whether its API key is available and returns the appropriate tool.

```ts
// src/index.ts
import type { ResolveToolsContext, ResolveToolsResult } from '@pandorakit/sdk/tools'
import { resolveSearchTool } from './resolve'

export const tools = []

export async function resolveTools(ctx: ResolveToolsContext): Promise<ResolveToolsResult> {
  const preferred = ctx.pluginConfig?.searchBackend as string | undefined
  const result = resolveSearchTool({ preferred, env: ctx.env })
  return { tools: result.tool ? [result.tool] : [], alerts: result.alerts }
}
```

A plugin can have both a static `tools` array and a `resolveTools` function. All tools â€” static and dynamically resolved â€” are loaded when the plugin is active.

The `resolveTools` function receives a `ResolveToolsContext`:

```ts
interface ResolveToolsContext {
  pluginConfig: Record<string, unknown>  // validated plugin config
  env: Record<string, string | undefined>
}
```

Return a `ResolveToolsResult` â€” an object with `tools` (array of `Tool`) and optional `alerts`:

```ts
interface ResolveToolsResult {
  tools: Tool[]
  alerts?: Alert[]
}
```

### Alerts

`resolveTools` can return diagnostic alerts alongside tools:

```ts
export async function resolveTools({ env }: ResolveToolsContext): Promise<ResolveToolsResult> {
  if (!env.MY_API_KEY) {
    return {
      tools: [],
      alerts: [{ level: 'warning', message: 'API key not set' }],
    }
  }
  return {
    tools: [myTool],
    alerts: [{ level: 'info', message: 'Using MyService' }],
  }
}
```

Alerts are stored at load time and surfaced in the Plugins page and the `/api/plugins` endpoint. Levels are `'info'` (informational) and `'warning'` (actionable).

## Reference

All types are exported from `@pandorakit/sdk/tools`:

```ts
import type {
  ResolveToolsContext,
  ResolveToolsResult,
  ToolAnnotations,
  Tool,
} from '@pandorakit/sdk/tools'
```

### `Tool`

The standard tool export interface. All plugins use this â€” no framework dependencies required.

| Field | Type | Required | Description |
|---|---|---|---|
| `id` | `string` | Yes | Unique tool identifier (within the plugin) |
| `name` | `string` | Yes | Human-readable display name |
| `description` | `string` | Yes | What the tool does (shown to the LLM) |
| `parameters` | `Record<string, unknown>` | No | JSON Schema for input validation |
| `execute` | `(input, context: { env }) => Promise` | Yes | The tool's handler function |
| `timeout` | `number` | No | Execution timeout in milliseconds. Defaults to `60000` (60 s) |
| `annotations` | `ToolAnnotations` | No | MCP-compatible UI hints |

### `ToolAnnotations`

```ts
interface ToolAnnotations {
  title?: string              // Display title in UI
  readOnlyHint?: boolean      // Safe to run without confirmation
  destructiveHint?: boolean   // May perform irreversible operations
  idempotentHint?: boolean    // Same args = no additional effect
}
```

### `EnvVarDescriptor`

Describes an environment variable a plugin depends on (declared in the manifest's `envVars`):

| Field | Type | Required | Description |
|---|---|---|---|
| `name` | `string` | Yes | Environment variable name |
| `required` | `boolean` | No | Whether this variable is required. Defaults to `true` |

### `ConfigFieldDescriptor`

Describes a config field for UI rendering (declared in the manifest's `configFields`):

| Field | Type | Required | Description |
|---|---|---|---|
| `key` | `string` | Yes | Field key in config object |
| `label` | `string` | Yes | Human-readable label |
| `type` | `'text' \| 'number' \| 'password' \| 'enum'` | Yes | Input type. `'enum'` renders a dropdown. |
| `required` | `boolean` | No | Whether this field is required |
| `placeholder` | `string` | No | Placeholder text |
| `description` | `string` | No | Help text shown below the input |
| `options` | `{ value: string; label: string }[]` | No | Allowed values for `type: 'enum'` |
