# Build a Tool

Tools are standalone npm packages that run in Node.js with full runtime access.

<Steps>

### Set up the package

A tool package follows this structure:

<FileTree>
  <FileTree.Folder name="pandora-tools-weather" defaultOpen>
    <FileTree.File name="package.json" />
    <FileTree.File name="pandora.manifest.json" />
    <FileTree.Folder name="src" defaultOpen>
      <FileTree.File name="index.ts" />
      <FileTree.File name="weather.ts" />
      <FileTree.File name="weather.test.ts" />
    </FileTree.Folder>
    <FileTree.File name="tsconfig.json" />
  </FileTree.Folder>
</FileTree>

The `package.json` declares `@pandora/core` as a dev dependency (only type-only imports are used at runtime):

```json
{
  "name": "pandora-tools-weather",
  "type": "module",
  "exports": { ".": "./src/index.ts" },
  "devDependencies": {
    "@pandora/core": "*"
  }
}
```

### Define the tool

Export a `ToolExport` object — a plain object with metadata, JSON Schema parameters, and an `execute` function:

```ts
// src/weather.ts
import type { ToolExport } from '@pandora/core/tools'

export const weatherTool: ToolExport = {
  id: 'weather',
  name: 'Weather',
  description: 'Get current weather for a city',
  parameters: {
    type: 'object',
    properties: {
      city: { type: 'string', description: 'City name' },
    },
    required: ['city'],
  },
  annotations: {
    readOnlyHint: true,
    destructiveHint: false,
    idempotentHint: true,
  },
  execute: async (input: { city: string }, { env }) => {
    const key = env.OPENWEATHER_API_KEY
    const res = await fetch(
      `https://api.openweathermap.org/data/2.5/weather?q=${input.city}&units=metric&appid=${key}`
    )
    return res.json()
  },
}
```

- `ToolExport` is a plain object — no framework dependencies at runtime. Only type-only imports from `@pandora/core` (erased by the TypeScript compiler).
- The `execute` function receives `{ env }` as its second argument, giving tools access to environment variables.
- **`annotations`** are MCP-compatible hints that help the UI decide whether to auto-approve or prompt for confirmation. A tool marked `readOnlyHint: true` and `destructiveHint: false` can be auto-approved.

### Export the plugin descriptor

Each package default-exports a `ToolPlugin` descriptor with metadata and a `tools` array listing the tool definitions:

```ts
// src/index.ts
import type { ToolPlugin } from '@pandora/core/tools'
import { weatherTool } from './weather'

export default {
  id: 'tools-weather',
  name: 'Weather',
  schemaVersion: 1,
  envVars: [{ name: 'OPENWEATHER_API_KEY' }],
  configFields: [
    {
      key: 'units',
      label: 'Units',
      type: 'enum',
      description: 'Temperature units',
      options: [
        { value: 'metric', label: 'Metric (°C)' },
        { value: 'imperial', label: 'Imperial (°F)' },
        { value: 'standard', label: 'Standard (K)' },
      ],
    },
  ],
  tools: [weatherTool],
} satisfies ToolPlugin
```

- **`name`** — human-readable display name for the UI.
- **`tools`** — array of `ToolExport` objects. Pandora wraps them into Mastra tools automatically when loading plugins.
- **`envVars`** — environment variables the plugin needs, as `{ name, required? }` descriptors. Variables default to required; set `required: false` for optional ones. The settings UI shows whether they're configured.
- **`configFields`** — describes user-configurable settings beyond `enabled`. Pandora validates input and renders fields in the settings UI from these descriptors.
- If `configFields` includes required fields and the user hasn't configured them, the plugin is skipped. Plugins with only optional config fields load with defaults automatically.

### Register it

Install your package in the Pandora instance and register it in `plugins.ts`:

```ts
import weather from 'pandora-tools-weather'
import { registerToolPlugin } from './tools'

registerToolPlugin(weather)
```

### Enable it

Once registered, users control your plugin in **Settings → Tools**:

- Toggle the plugin on/off
- Configure settings from your `configFields`
- Optionally enable **Require Approval** to prompt before any of the plugin's tools run

A plugin with required `configFields` is skipped until the user configures them. Plugins with only optional config fields load with defaults automatically.

</Steps>

## Dynamic Tool Resolution

Some tools can't be defined statically — they depend on which API keys are available or user configuration. For these cases, `ToolPlugin` supports an async `resolveTools` hook that runs at load time and returns `ToolExport` objects dynamically.

This is the mechanism behind `@pandora/tools-websearch`: it checks which search API keys are available and returns the appropriate backend as a `ToolExport`.

```ts
// src/index.ts
import { resolveSearchTool } from './resolve'

export const tools: [] = []

export async function resolveTools({ pluginConfig, env }) {
  const preferred = pluginConfig?.searchBackend as string | undefined
  const result = resolveSearchTool({ preferred, env })
  return { tools: result.tool ? [result.tool] : [], alerts: result.alerts }
}
```

A plugin can have both static `tools` and a `resolveTools` hook. All tools — static and dynamically resolved — are loaded when the plugin is active. Resolved tools go through `bindToolExport()` + `registerManifest()`, so they get manifests, timeouts, and Zod schema wrapping just like static tools.

The `resolveTools` hook receives a `ResolveToolsContext`:

```ts
interface ResolveToolsContext {
  pluginConfig: Record<string, unknown>  // validated plugin config
  env: Record<string, string | undefined>
}
```

Return a `ResolveToolsResult` — an object with `tools` (array of `ToolExport`) and optional `alerts`:

```ts
interface ResolveToolsResult {
  tools: ToolExport[]
  alerts?: Alert[]
}
```

### Alerts

`resolveTools` can return diagnostic alerts alongside tools:

```ts
async resolveTools({ env }) {
  if (!env.MY_API_KEY) {
    return {
      tools: [],
      alerts: [{ level: 'warning', message: 'API key not set' }],
    }
  }
  return {
    tools: [myToolExport],
    alerts: [{ level: 'info', message: 'Using MyService' }],
  }
}
```

Alerts are stored at load time and surfaced in the settings UI and the `/api/tools` endpoint. Levels are `'info'` (informational) and `'warning'` (actionable).

## Reference

All types are exported from `@pandora/core/tools`:

```ts
import type {
  ConfigFieldDescriptor,
  EnvVarDescriptor,
  ResolveToolsContext,
  ResolveToolsResult,
  ToolAnnotations,
  ToolExport,
  ToolPlugin,
} from '@pandora/core/tools'
```

### `ToolExport`

The standard tool export interface. All plugins use this — no framework dependencies required.

| Field | Type | Required | Description |
|---|---|---|---|
| `id` | `string` | Yes | Unique tool identifier |
| `name` | `string` | Yes | Human-readable display name |
| `description` | `string` | Yes | What the tool does (shown to the LLM) |
| `parameters` | `Record<string, unknown>` | No | JSON Schema for input validation |
| `execute` | `(input, context: { env }) => Promise` | Yes | The tool's handler function |
| `timeout` | `number` | No | Execution timeout in milliseconds. Defaults to `60000` (60 s) |
| `annotations` | `ToolAnnotations` | No | MCP-compatible UI hints |

### `ToolAnnotations`

```ts
interface ToolAnnotations {
  title?: string              // Display title in UI
  readOnlyHint?: boolean      // Safe to run without confirmation
  destructiveHint?: boolean   // May perform irreversible operations
  idempotentHint?: boolean    // Same args = no additional effect
}
```

### `ToolPluginConfig`

The validated user configuration passed to tool definitions:

```ts
interface ToolPluginConfig {
  enabled: boolean
  [key: string]: unknown // custom fields from configFields
}
```

### `ToolPlugin`

The descriptor your package default-exports:

| Field | Type | Required | Description |
|---|---|---|---|
| `id` | `string` | Yes | Plugin identifier, e.g. `'tools-datetime'` |
| `name` | `string` | Yes | Human-readable display name, e.g. `'Date & Time'` |
| `schemaVersion` | `number` | Yes | Must be `1` |
| `envVars` | `EnvVarDescriptor[]` | No | Environment variable descriptors |
| `configFields` | `ConfigFieldDescriptor[]` | No | UI config field descriptors |
| `tools` | `ToolExport[]` | Yes | Tool exports — plain objects with metadata and execute function |
| `resolveTools` | `(ctx: ResolveToolsContext) => Promise<ResolveToolsResult>` | No | Async hook for dynamic tool resolution. Returns `ToolExport[]` and optional alerts. See [Dynamic Tool Resolution](#dynamic-tool-resolution). |

### `EnvVarDescriptor`

Describes an environment variable a plugin depends on:

| Field | Type | Required | Description |
|---|---|---|---|
| `name` | `string` | Yes | Environment variable name |
| `required` | `boolean` | No | Whether this variable is required. Defaults to `true` |
