# Build a Storage Plugin

Every storage plugin is a standalone npm package that default-exports a `StoragePlugin` descriptor. Your factory receives environment variables and returns three layers: a [Mastra storage adapter](https://mastra.ai/docs/memory/storage), a config store, and an auth store.

<Callout type="info">
  Pandora's storage layer depends on Mastra for thread and memory persistence. You can only build a provider for databases that Mastra already supports — see [Mastra's storage docs](https://mastra.ai/docs/memory/storage) for the current list.
</Callout>

## Built-in Helpers

Most providers don't need to implement `ConfigStore` or `AuthStore` from scratch. Pandora ships reusable implementations for SQL and Redis-like databases.

<Tabs items={['SQL', 'Redis']}>
  <Tabs.Tab>
    For SQL databases (SQLite/LibSQL, PostgreSQL, MSSQL). You provide an `execute` function wrapping your driver:

    ```ts
    import { SQLConfigStore, SQLAuthStore } from '@pandora/core/storage'

    const execute = async (sql: string, params?: unknown[]) => {
      const result = await myDriver.query(sql, params)
      return result.rows
    }

    const config = new SQLConfigStore<Config>(execute, 'postgres')
    const auth = new SQLAuthStore(execute, 'postgres')
    ```

    The `execute` function receives raw SQL and an optional params array. It must return an array of row objects. The dialect argument (`'sqlite'` | `'postgres'` | `'mssql'`) handles differences in placeholders, upsert strategy, and JSON storage.
  </Tabs.Tab>
  <Tabs.Tab>
    For Redis-like key-value stores. You provide an object with `get`, `set`, and `del` methods:

    ```ts
    import { RedisConfigStore, RedisAuthStore } from '@pandora/core/storage'

    const ops = {
      get: (key) => myRedis.get(key),
      set: (key, value, opts) => myRedis.set(key, value, opts),
      del: (key) => myRedis.del(key),
      smembers: (key) => myRedis.smembers(key),
      sadd: (key, ...members) => myRedis.sadd(key, ...members),
      srem: (key, ...members) => myRedis.srem(key, ...members),
    }

    const config = new RedisConfigStore<Config>(ops)
    const auth = new RedisAuthStore(ops)
    ```
  </Tabs.Tab>
</Tabs>

If neither fits your database, implement `ConfigStore` and `AuthStore` directly — see the [Reference](#reference) section below for interface contracts.

## Creating a Plugin

Let's build a PostgreSQL plugin as an example.

<Steps>

### Create the package

<FileTree>
  <FileTree.Folder name="pandora-storage-postgres" defaultOpen>
    <FileTree.File name="package.json" />
    <FileTree.Folder name="src" defaultOpen>
      <FileTree.File name="index.ts" />
    </FileTree.Folder>
    <FileTree.File name="tsconfig.json" />
  </FileTree.Folder>
</FileTree>

### Set up `package.json`

```json
{
  "name": "pandora-storage-postgres",
  "version": "0.0.1",
  "type": "module",
  "exports": {
    ".": "./src/index.ts"
  },
  "dependencies": {
    "@mastra/pg": "latest",
    "pg": "^8.16.0"
  },
  "peerDependencies": {
    "@pandora/core": "*"
  }
}
```

Key conventions:
- `@pandora/core` is a peer dependency
- Include the `@mastra/*` adapter for your database
- Default-export a `StoragePlugin` descriptor from the package entry point

### Implement the plugin

Your entry point default-exports a `StoragePlugin` with metadata and a factory that creates three things: the Mastra adapter, a config store, and an auth store.

```ts
// src/index.ts
import { Pool } from 'pg'
import { PostgresStore } from '@mastra/pg'
import type { Config, StorageFactory, StoragePlugin } from '@pandora/core/storage'
import { SQLAuthStore, SQLConfigStore } from '@pandora/core/storage'

const createStorage: StorageFactory = async (env) => {
  const pool = new Pool({
    connectionString: env.DATABASE_URL,
  })

  // Mastra adapter — handles thread and memory persistence
  const mastra = new PostgresStore({
    id: 'pandora-postgres',
    pool,
  })

  // Pandora config and auth stores — use SQL helpers
  const config = new SQLConfigStore<Config>(async (sql, params) => {
    const result = await pool.query(sql, params)
    return result.rows
  }, 'postgres')

  const auth = new SQLAuthStore(async (sql, params) => {
    const result = await pool.query(sql, params)
    return result.rows
  }, 'postgres')

  return { mastra, config, auth, close: () => pool.end() }
}

export default {
  id: 'storage-postgres',
  name: 'PostgreSQL',
  schemaVersion: 1,
  envVars: ['DATABASE_URL'],
  factory: createStorage,
} satisfies StoragePlugin
```

A few things to notice:

- **Mastra adapter** — the `@mastra/pg` package handles thread storage, memory, and workflows. This is why you need Mastra to support your database first.
- **SQL helpers** handle all table creation, parameterization, and dialect differences for Pandora's own config and auth. You just provide an `execute` function wrapping your driver.
- **`init()`** on each store is called automatically by Pandora after `createStorage` returns — you don't need to call it yourself.
- **`close()`** — if your database driver holds a persistent connection (connection pool, TCP socket, etc.), return a `close` function that tears it down. Pandora calls it on shutdown. HTTP-based providers (like Upstash) can omit it.

### Register and test it

Install your package in the Pandora instance, register it in `plugins.ts`, and start:

```ts
import postgres from 'pandora-storage-postgres'
import { registerStoragePlugin } from './storage'

registerStoragePlugin(postgres)
```

```bash
STORAGE_PROVIDER=storage-postgres DATABASE_URL=postgres://localhost/pandora bun run dev
```

</Steps>

## Reference

All types are exported from `@pandora/core/storage`:

```ts
import type {
  AuthStore,
  Config,
  ConfigFieldDescriptor,
  ConfigStore,
  StorageFactory,
  StoragePlugin,
  StorageResult,
} from '@pandora/core/storage'

import {
  SQLConfigStore,
  SQLAuthStore,
  RedisConfigStore,
  RedisAuthStore,
} from '@pandora/core/storage'
```

### `StoragePlugin`

The descriptor your package default-exports:

| Field | Type | Required | Description |
|---|---|---|---|
| `id` | `string` | Yes | Plugin identifier, e.g. `'storage-postgres'` |
| `name` | `string` | Yes | Human-readable display name, e.g. `'PostgreSQL'` |
| `schemaVersion` | `number` | Yes | Must be `1` |
| `envVars` | `EnvVarDescriptor[]` | No | Environment variable descriptors |
| `configFields` | `ConfigFieldDescriptor[]` | No | UI config field descriptors |
| `factory` | `StorageFactory` | Yes | Creates storage instances from env vars |

### `StorageFactory`

The factory function inside a `StoragePlugin`:

```ts
type StorageFactory = (
  env: Record<string, string | undefined>,
  bindings?: unknown,
) => Promise<StorageResult>
```

| Parameter | Description |
|---|---|
| `env` | Environment variables |
| `bindings` | Cloudflare Workers bindings, if applicable |

### `StorageResult`

The object your factory must return:

```ts
interface StorageResult {
  /** Mastra storage for memory, workflows, etc. */
  mastra: MastraCompositeStore
  /** Pandora config storage */
  config: ConfigStore<Config>
  /** Pandora auth storage */
  auth: AuthStore
  /** Close all connections (optional cleanup on shutdown) */
  close?(): Promise<void>
}
```

### `ConfigStore<T>`

```ts
interface ConfigStore<T = unknown> {
  /** Get the current config, or null if not set */
  get(): Promise<T | null>

  /** Save the config (full replacement) */
  set(config: T): Promise<void>

  /** Delete the config */
  delete(): Promise<void>

  /** Initialize tables/collections if needed (optional) */
  init?(): Promise<void>

  /** Close connections (optional cleanup) */
  close?(): Promise<void>
}
```

Implementation requirements:
- `get()` must return `null` (not throw) if no config exists yet
- `set()` performs a full replacement (upsert)
- `delete()` should not throw if the config doesn't exist
- `init()` is called automatically by Pandora after `createStorage` returns — use it for schema setup

### `AuthStore`

```ts
interface AuthStore {
  init(): Promise<void>
  getCredential(): Promise<PasswordCredential | null>
  setCredential(credential: PasswordCredential): Promise<void>
  setCredentialIfNotExists(credential: PasswordCredential): Promise<boolean>
  createSession(session: Session): Promise<void>
  getSession(tokenHash: string): Promise<Session | null>
  deleteSession(tokenHash: string): Promise<void>
  deleteAllSessions(): Promise<void>
  listSessions(): Promise<Session[]>
  createRefreshToken(token: RefreshToken): Promise<void>
  getRefreshToken(tokenHash: string): Promise<RefreshToken | null>
  deleteRefreshToken(tokenHash: string): Promise<void>
  deleteAllRefreshTokens(): Promise<void>
  markRefreshTokenUsed(tokenHash: string): Promise<void>
  close?(): Promise<void>
}
```

Implementation requirements:
- `init()` is called automatically — use it to create tables/collections
- `getCredential()` and `getSession()` must return `null` if not found
- `setCredentialIfNotExists()` must be atomic — return `true` if set, `false` if one already exists

### SQL Helpers

`SQLConfigStore` and `SQLAuthStore` accept an `execute` function and a dialect:

```ts
type SQLDialect = 'sqlite' | 'postgres' | 'mssql'

new SQLConfigStore<Config>(execute, dialect)
new SQLAuthStore(execute, dialect)
```

The `execute` function signature:

```ts
(sql: string, params?: unknown[]) => Promise<Record<string, unknown>[]>
```

Supported dialects handle differences in:
- Table creation syntax
- Parameterized query placeholders (`?`, `$1`, `@p1`)
- Upsert strategy (`ON CONFLICT` vs `MERGE`)
- JSON storage (`TEXT`, `JSONB`, `NVARCHAR(MAX)`)

`SQLAuthStore` creates three tables: `pandora_auth_credentials`, `pandora_auth_sessions`, `pandora_auth_refresh_tokens`.

### Redis Helpers

`RedisConfigStore` and `RedisAuthStore` accept an operations object:

```ts
// Config store needs get/set/del
new RedisConfigStore<Config>({ get, set, del })

// Auth store additionally needs set members
new RedisAuthStore({ get, set, del, smembers, sadd, srem })
```

Config is stored under the key `pandora:config`. Auth uses key prefixes: `pandora:auth:credential`, `pandora:auth:session:*`, `pandora:auth:refresh:*`.
