# Bring Your Own UI

Build a custom chat frontend against Pandora's [AI SDK-compatible](https://ai-sdk.dev/docs/ai-sdk-ui/chatbot) streaming API.

## Chat API Format

### Request

```
POST /api/chat
Content-Type: application/json
```

The request body sends the latest user message parts and an optional thread ID:

```json
{
  "parts": [{ "type": "text", "text": "Hello" }],
  "threadId": "abc123"
}
```

| Field | Type | Description |
|---|---|---|
| `parts` | `array` | **Required.** User message parts — at minimum `[{ "type": "text", "text": "..." }]` |
| `threadId` | `string` | Optional. Omit to start a new thread (server generates a UUID). |

### Response

The response is a `UIMessageStreamResponse` — a Server-Sent Events stream that the AI SDK client libraries can consume directly. The stream emits chunks for text, tool calls, reasoning, sources, and other part types.

The response includes an `X-Thread-Id` header with the thread ID (useful when the server creates a new thread).

## Thread Endpoints

### List threads

```
GET /api/threads
```

Returns threads ordered by most recently updated:

```json
{
  "threads": [
    {
      "id": "abc123",
      "title": "Getting started with Pandora",
      "createdAt": "2025-01-01T00:00:00.000Z",
      "updatedAt": "2025-01-01T00:01:00.000Z"
    }
  ],
  "total": 1,
  "page": 0,
  "perPage": 100,
  "hasMore": false,
  "activeStreamIds": ["abc123"]
}
```

| Field | Type | Description |
|---|---|---|
| `threads` | `array` | Thread objects ordered by `updatedAt` descending |
| `total` | `number` | Total thread count |
| `page` | `number` | Current page index |
| `perPage` | `number \| false` | Page size |
| `hasMore` | `boolean` | Whether more pages exist |
| `activeStreamIds` | `string[]` | Thread IDs that are currently streaming a response. Always `[]` in serverless mode. |

### Get thread with messages

```
GET /api/threads/:id
```

Returns the thread metadata and its message history:

```json
{
  "thread": {
    "id": "abc123",
    "title": "Getting started with Pandora",
    "createdAt": "2025-01-01T00:00:00.000Z",
    "updatedAt": "2025-01-01T00:01:00.000Z"
  },
  "messages": [
    {
      "id": "msg-1",
      "role": "user",
      "content": { "format": 2, "parts": [{ "type": "text", "text": "Hello" }] },
      "createdAt": "2025-01-01T00:00:00.000Z"
    }
  ]
}
```

### Resume stream

```
GET /api/chat/:chatId/stream
```

Reconnects to an in-progress stream for the given thread. This is the server-side half of the AI SDK's [resumable streams](https://ai-sdk.dev/docs/ai-sdk-ui/resumable-streams) protocol — `useChat` calls this automatically when `resume: true` is set.

| Status | Meaning |
|---|---|
| `200` | Active stream found — returns the buffered SSE stream from the point the client connects |
| `204` | No active stream (already finished, or serverless mode) |

In persistent server mode, the server buffers each SSE stream in memory so that clients can reconnect mid-stream. The buffer is cleaned up 60 seconds after the stream completes. In serverless mode, this endpoint always returns `204` since there is no shared memory between request isolates.

## Using AI SDK React

The recommended approach for React frontends is [`useChat`](https://ai-sdk.dev/docs/reference/ai-sdk-ui/use-chat) with [`DefaultChatTransport`](https://ai-sdk.dev/docs/ai-sdk-ui/transport):

```tsx
import { useChat } from '@ai-sdk/react'
import { DefaultChatTransport } from 'ai'

export default function Chat({ threadId }: { threadId?: string }) {
  const { messages, sendMessage, status } = useChat({
    transport: new DefaultChatTransport({
      api: 'http://localhost:4111/api/chat',
      prepareSendMessagesRequest: ({ messages }) => {
        const lastMessage = messages.at(-1)
        const parts = lastMessage?.role === 'user' ? lastMessage.parts : []
        return { body: { parts, threadId } }
      },
    }),
  })

  return (
    <div>
      {messages.map((message) => (
        <div key={message.id}>
          <strong>{message.role}:</strong>
          {message.parts.map((part, i) => {
            if (part.type === 'text') return <p key={i}>{part.text}</p>
            return null
          })}
        </div>
      ))}

      <form onSubmit={(e) => {
        e.preventDefault()
        const input = new FormData(e.currentTarget).get('message') as string
        sendMessage({ text: input })
        e.currentTarget.reset()
      }}>
        <input name="message" placeholder="Send a message..." />
        <button type="submit">Send</button>
      </form>
    </div>
  )
}
```

### Key Properties

| Property | Description |
|---|---|
| `messages` | Array of `UIMessage` objects with `role`, `id`, and `parts` |
| `sendMessage` | Send a new user message — accepts `{ text: string }` |
| `status` | `'ready'` \| `'submitted'` \| `'streaming'` \| `'error'` |

### Resumable Streams

When a user navigates away from a streaming thread (e.g., switching threads in a sidebar), the `useChat` instance unmounts and the stream is lost. Pandora supports the AI SDK's [resumable streams](https://ai-sdk.dev/docs/ai-sdk-ui/resumable-streams) protocol so that navigating back to the thread picks up where it left off.

To enable resume, pass `resume: true` to `useChat` and add `prepareReconnectToStreamRequest` to the transport (for auth headers on the reconnect GET request):

```tsx
const transport = new DefaultChatTransport({
  api: 'http://localhost:4111/api/chat',
  prepareSendMessagesRequest: ({ messages }) => {
    const lastMessage = messages.at(-1)
    const parts = lastMessage?.role === 'user' ? lastMessage.parts : []
    return { body: { parts, threadId } }
  },
  prepareReconnectToStreamRequest: () => {
    const token = getToken()
    return token ? { headers: { Authorization: `Bearer ${token}` } } : {}
  },
})

const { messages, sendMessage, status } = useChat({
  id: threadId,
  transport,
  resume: true,
})
```

On mount, `useChat` sends `GET /api/chat/{threadId}/stream`. If an active stream exists, it resumes seamlessly; otherwise the server returns `204` and the client shows persisted messages. In serverless mode, the GET always returns `204` — this is harmless.

You can use the `activeStreamIds` field from `GET /api/threads` to show streaming indicators in your UI. Poll the threads endpoint while `activeStreamIds` is non-empty to detect when streams complete:

```ts
const { data } = useQuery({
  queryKey: ['threads'],
  queryFn: () => fetch('/api/threads').then(r => r.json()),
  refetchInterval: (query) => {
    const ids = query.state.data?.activeStreamIds
    return ids?.length ? 3000 : false
  },
})
```

## Message Part Types Reference

Each message has a `parts` array. Here are all the part types the Pandora API can emit (see [UIMessage reference](https://ai-sdk.dev/docs/reference/ai-sdk-core/ui-message) for full type definitions):

| Part Type | TypeScript Type | Fields | When Emitted |
|---|---|---|---|
| `text` | `TextUIPart` | `text`, `state?` (`'streaming'` \| `'done'`) | Always — LLM text output |
| `tool-{name}` | `ToolUIPart` | `toolCallId`, `state`, `input`, `output`, `errorText` | When the agent calls a tool |
| `reasoning` | `ReasoningUIPart` | `text`, `state?`, `providerMetadata?` | When the model supports reasoning |
| `source-url` | `SourceUrlUIPart` | `sourceId`, `url`, `title?`, `providerMetadata?` | When the model provides citations |
| `file` | `FileUIPart` | `mediaType`, `url`, `filename?` | When the response includes files |
| `step-start` | `StepStartUIPart` | — | At the start of each agent step |

## Rendering Parts

The standard pattern for rendering message parts:

```tsx
function MessageParts({ message }: { message: UIMessage }) {
  return (
    <>
      {message.parts.map((part, i) => {
        switch (part.type) {
          case 'text':
            return <p key={i}>{part.text}</p>
          case 'reasoning':
            return <details key={i}><summary>Thinking...</summary>{part.text}</details>
          case 'source-url':
            return <a key={i} href={part.url}>{part.title || part.url}</a>
          default:
            // Tool parts have type like "tool-current-time"
            if (part.type.startsWith('tool-')) {
              return (
                <pre key={i}>
                  Tool: {part.type} ({part.state})
                  {part.state === 'output-available' && JSON.stringify(part.output, null, 2)}
                </pre>
              )
            }
            return null
        }
      })}
    </>
  )
}
```

## Without AI SDK

For non-React frontends (Vue, Svelte, vanilla JS), you can consume the stream manually with `fetch`:

```js
const response = await fetch('http://localhost:4111/api/chat', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    parts: [{ type: 'text', text: 'Hello' }],
    threadId: 'optional-thread-id'
  })
})

const reader = response.body.getReader()
const decoder = new TextDecoder()

// Read the thread ID from the response header
const threadId = response.headers.get('X-Thread-Id')

while (true) {
  const { done, value } = await reader.read()
  if (done) break
  const chunk = decoder.decode(value, { stream: true })
  // Parse SSE lines: each line is a JSON-encoded stream part
  console.log(chunk)
}
```

For a full implementation, see the AI SDK's [`readUIMessageStream`](https://ai-sdk.dev/docs/ai-sdk-ui/reading-ui-message-streams) utility which handles parsing the stream format:

```ts
import { readUIMessageStream } from 'ai'

const response = await fetch('http://localhost:4111/api/chat', { /* ... */ })

for await (const message of readUIMessageStream({ stream: response.body })) {
  // message is a UIMessage that updates as parts stream in
  console.log(message)
}
```

## CORS

The core server enables CORS for all origins by default via Hono's `cors()` middleware. If you need to restrict origins in production, update the CORS config in `packages/core/src/index.ts`:

```ts
app.use('*', cors({ origin: 'https://your-ui.example.com' }))
```
