# Custom Tools

Tools give your AI new capabilities. This guide walks you through creating one.

## Example: Weather tool

Let's build a weather tool that fetches current conditions.

<Steps>

### Create the file

Create `packages/pandora/src/tools/weather.ts`:

```typescript
import { tool } from "ai";
import { z } from "zod";
import { defineTool } from "@pandora/core";

export default defineTool({
  name: "weather",

  factory: (config) => ({
    name: "weather",

    tool: tool({
      description: "Get current weather for a location",

      parameters: z.object({
        location: z.string().describe("City name, e.g. 'London' or 'New York'"),
      }),

      execute: async ({ location }) => {
        const apiKey = (config as { apiKey?: string })?.apiKey;

        const response = await fetch(
          `https://api.weatherapi.com/v1/current.json?key=${apiKey}&q=${location}`
        );
        const data = await response.json();

        return `Weather in ${location}: ${data.current.temp_c}°C, ${data.current.condition.text}`;
      },
    }),
  }),
});
```

### Add to config

Edit `config.jsonc`:

```jsonc
"tools": {
  "weather": {
    "apiKey": "your-weather-api-key"
  }
}
```

### Done!

Restart Pandora. Now you can ask "What's the weather in Tokyo?"

</Steps>

## How it works

```typescript
export default defineTool({
  name: "weather",        // Unique name, matches config key

  factory: (config) => ({ // Config from config.jsonc
    name: "weather",      // Name for logging
    tool: tool({ ... }),  // AI SDK tool definition
  }),
});
```

The `factory` function receives your tool's config (e.g., `{ apiKey: "..." }`).

The `tool()` function comes from Vercel AI SDK and defines:
- `description` — What the AI sees when deciding to use this tool
- `parameters` — Zod schema for input validation
- `execute` — The actual implementation

## Returning data

Tools can return:

**Strings** — Simple text results

```typescript
execute: async ({ location }) => {
  return `Weather in ${location}: 72°F, Sunny`;
}
```

**Objects** — Structured data (AI will summarize)

```typescript
execute: async ({ location }) => {
  return {
    location,
    temperature: 72,
    conditions: "Sunny",
    humidity: 45,
  };
}
```

---

## Restricting to specific agents

By default, tools are available to all agents. To restrict:

```typescript
factory: (config) => ({
  name: "codeRunner",
  tool: tool({ ... }),

  // Only the coder agent can use this
  agents: ["coder"],
}),
```

## Tools without config

If your tool doesn't need configuration:

```typescript
import { tool } from "ai";
import { z } from "zod";
import { defineTool } from "@pandora/core";

export default defineTool({
  name: "coinFlip",

  factory: () => ({
    name: "coinFlip",

    tool: tool({
      description: "Flip a coin",
      parameters: z.object({}),

      execute: async () => {
        return Math.random() > 0.5 ? "Heads" : "Tails";
      },
    }),
  }),
});
```

Config:

```jsonc
"tools": {
  "coinFlip": {}
}
```

## Error handling

<Callout type="info">
Throw errors to signal failure. The AI will see the error message and can adjust its approach.
</Callout>

```typescript
execute: async ({ ticker }) => {
  const response = await fetch(`https://api.stocks.com/${ticker}`);

  if (!response.ok) {
    throw new Error(`Could not find stock: ${ticker}`);
  }

  const data = await response.json();
  return `${ticker}: $${data.price}`;
}
```

## Search tool dual-registration

Some tools serve double duty: they work as standalone tools _and_ as search backends
for the `webSearchTool` subagent. This uses the dual-registration pattern.

Here is how `tavily-search.ts` works in practice:

```typescript
import { tavilySearch } from "@tavily/ai-sdk";
import { defineTool, defineSearchTool, type ToolConfig } from "@pandora/core";

// Register as a regular tool (available to all agents)
export default defineTool({
  name: "tavilySearch",
  factory: (config?: ToolConfig) => {
    const apiKey = config?.apiKey as string | undefined;
    if (!apiKey) {
      throw new Error("tavilySearch tool requires 'apiKey' in config");
    }

    return {
      name: "tavilySearch",
      tool: tavilySearch({ apiKey }),
    };
  },
});

// ALSO register as a search backend (for webSearchTool subagent)
defineSearchTool({
  name: "tavilySearch",
  description: "Tavily AI-powered web search with high-quality results",
  factory: (config?: ToolConfig) => {
    const apiKey = config?.apiKey as string | undefined;
    if (!apiKey) {
      throw new Error("tavilySearch requires 'apiKey' in config");
    }
    return tavilySearch({ apiKey });
  },
});
```

The `defineSearchTool()` registration lets the `webSearchTool` subagent find and use
this tool as its search backend. The subagent's config specifies which backend to use:

```jsonc
"agents": {
  "webSearchTool": {
    "model": "anthropic/claude-sonnet-4.5",
    "searchBackend": "tavilySearch"  // Picks from registered search tools
  }
},
"tools": {
  "tavilySearch": { "apiKey": "tvly-..." }
}
```

### The factory pattern

Every tool's `factory` function receives its config object from `config.ai.tools[name]`.
For example, if your config has:

```jsonc
"tools": {
  "weather": { "apiKey": "abc123", "units": "metric" }
}
```

Then `factory(config)` receives `{ "apiKey": "abc123", "units": "metric" }`.

This lets tools read API keys, toggle features, or accept any configuration
without changing the framework.

## Accessing request context

Tools can access per-request data (conversation ID, channel name) via `requestContext`:

```typescript
import { tool } from "ai";
import { z } from "zod";
import { defineTool, requestContext } from "@pandora/core";

export default defineTool({
  name: "myTool",

  factory: () => ({
    name: "myTool",

    tool: tool({
      description: "A tool that uses request context",
      parameters: z.object({ query: z.string() }),

      execute: async ({ query }) => {
        const ctx = requestContext.getStore();

        // ctx?.conversationId — current conversation
        // ctx?.channelName — originating channel (e.g. "telegram", "web")

        return `Running in conversation ${ctx?.conversationId}`;
      },
    }),
  }),
});
```

The context is set automatically by the Gateway for each request and is concurrency-safe. See [Architecture: Request context](/reference/architecture#request-context) for how it works.

## Tips

- **Keep tools focused** — One tool, one purpose
- **Good descriptions** — Help the AI know when to use it
- **Validate inputs** — Zod schemas catch bad input early
- **Handle errors** — Throw meaningful error messages
- **Test incrementally** — Add one tool at a time
- **Dual-register search tools** — Use both `defineTool()` and `defineSearchTool()` if your tool is a search backend
