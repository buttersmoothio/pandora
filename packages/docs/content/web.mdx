# Web Channel

The web channel exposes an HTTP + WebSocket API that any frontend can connect to. Responses stream token-by-token over WebSocket.

## Setup

### Step 1: Choose an auth token

Pick a secret token that clients must provide to authenticate:

```
my-secret-token-here
```

### Step 2: Configure Pandora

Edit `config.jsonc`:

```jsonc
"channels": {
  "web": {
    "port": 3000,
    "token": "my-secret-token-here"
  }
}
```

Adding a channel to the config enables it. Remove it to disable.

| Field | Description |
|-------|-------------|
| `port` | HTTP/WebSocket port (default: `3000`) |
| `token` | Auth token that clients must provide |

### Step 3: Start Pandora

```bash
bun run start
```

You should see:

```
[Channel] [web] Server started url=http://localhost:3000 ws=ws://localhost:3000/ws
```

## WebSocket API (streaming)

Connect to `ws://localhost:3000/ws?token=YOUR_TOKEN` for real-time streaming.

### Send a message

```json
{ "type": "message", "content": "Hello, what can you do?" }
```

The server responds with a stream of deltas followed by a done signal:

```json
{ "type": "delta", "text": "I" }
{ "type": "delta", "text": " can" }
{ "type": "delta", "text": " help" }
...
{ "type": "done" }
```

### Clear conversation

```json
{ "type": "clear" }
```

Server responds:

```json
{ "type": "cleared" }
```

### Error handling

If something goes wrong, the server sends:

```json
{ "type": "error", "message": "Description of what went wrong" }
```

### Example (JavaScript)

```js
const ws = new WebSocket("ws://localhost:3000/ws?token=my-secret-token-here");

ws.onopen = () => {
  ws.send(JSON.stringify({ type: "message", content: "Hello!" }));
};

ws.onmessage = (event) => {
  const data = JSON.parse(event.data);
  switch (data.type) {
    case "delta":
      process.stdout.write(data.text); // Stream to terminal
      break;
    case "done":
      console.log("\n--- Response complete ---");
      break;
    case "error":
      console.error("Error:", data.message);
      break;
  }
};
```

### Watch a conversation (cross-channel streaming)

The web UI shows conversations from **all channels** (Telegram, web, etc.). To receive live streaming events for any conversation — including those from other channels — send a `watch` message:

```json
{ "type": "watch", "conversationId": "telegram-123456" }
```

While watching, you'll receive real-time events as they happen:

```json
{ "type": "user-message", "conversationId": "telegram-123456", "channelName": "telegram", "content": "Hello" }
{ "type": "delta", "conversationId": "telegram-123456", "text": "Hi" }
{ "type": "delta", "conversationId": "telegram-123456", "text": " there!" }
{ "type": "tool-call", "conversationId": "telegram-123456", "toolCallId": "...", "toolName": "...", "args": {} }
{ "type": "tool-result", "conversationId": "telegram-123456", "toolCallId": "...", "toolName": "...", "result": "..." }
{ "type": "done", "conversationId": "telegram-123456" }
```

To stop watching:

```json
{ "type": "unwatch" }
```

Watching automatically switches when you send a new `watch` — the previous subscription is replaced. The subscription is also cleaned up when the WebSocket disconnects.

## HTTP API (non-streaming)

For clients that don't need streaming, a simple REST endpoint is available.

### Send a message

```bash
curl -X POST http://localhost:3000/api/message \
  -H "Authorization: Bearer my-secret-token-here" \
  -H "Content-Type: application/json" \
  -d '{"content": "Hello!"}'
```

Response:

```json
{ "response": "Hello! How can I help you today?" }
```

### Clear conversation

```bash
curl -X POST http://localhost:3000/api/clear \
  -H "Authorization: Bearer my-secret-token-here"
```

Response:

```json
{ "ok": true }
```

## Authentication

All requests require the `token` configured in your channel config:

- **WebSocket** — Pass as query parameter: `?token=YOUR_TOKEN`
- **HTTP** — Pass as Bearer token: `Authorization: Bearer YOUR_TOKEN`

Unauthorized requests receive a `401` response and are not processed.

## Capabilities

| Capability | Supported |
|------------|-----------|
| Rich text (Markdown) | Yes |
| Streaming | Yes |
| Images | No |
| Files | No |
| Buttons | No |
| Max message length | Unlimited |

## Troubleshooting

### "Unauthorized" on connect

- Check that your token matches the `token` in `config.jsonc` exactly
- For WebSocket: ensure the token is URL-encoded in the query parameter
- For HTTP: ensure the `Authorization` header uses the `Bearer` prefix

### Port already in use

Change the `port` in your config, or stop whatever is using that port.

### WebSocket disconnects immediately

- Check the server logs for errors
- Ensure the upgrade path is `/ws` (not `/` or `/api/ws`)

## Security notes

- **Keep your token secret** — Anyone with it can use your AI agent
- **Keep your config private** — Don't commit `config.jsonc` to public repos
- **Localhost only by default** — Bun serves on `0.0.0.0`, so use a firewall or reverse proxy for production
