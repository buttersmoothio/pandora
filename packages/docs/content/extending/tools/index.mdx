# Building a Tool

Tool plugins are standalone packages that provide tools Pandora can use. Each plugin is automatically discovered via its `pandora.manifest.json` — no manual registration needed.

<Steps>

### Set up the package

A tool package follows this structure:

<FileTree>
  <FileTree.Folder name="weather" defaultOpen>
    <FileTree.File name="package.json" />
    <FileTree.File name="pandora.manifest.json" />
    <FileTree.Folder name="src" defaultOpen>
      <FileTree.File name="index.ts" />
      <FileTree.File name="weather.ts" />
    </FileTree.Folder>
    <FileTree.File name="tsconfig.json" />
  </FileTree.Folder>
</FileTree>

The `package.json` declares `@pandorakit/sdk` as a dev dependency (only type-only imports are used at runtime):

```json
{
  "name": "@pandorakit/weather",
  "type": "module",
  "exports": { ".": "./src/index.ts" },
  "devDependencies": {
    "@pandorakit/sdk": "*"
  }
}
```

### Write the manifest

The `pandora.manifest.json` declares your plugin's identity, capabilities, and requirements:

```json
{
  "manifestVersion": 1,
  "id": "@pandorakit/weather",
  "name": "Weather",
  "description": "Get current weather for any city",
  "author": "You",
  "version": "0.1.0",
  "license": "MIT",
  "pandora": ">=0.0.1",
  "provides": {
    "tools": {
      "entry": "./src/index.ts",
      "sandbox": "compartment",
      "permissions": {
        "network": ["api.openweathermap.org"],
        "env": ["OPENWEATHER_API_KEY"]
      }
    }
  },
  "envVars": [{ "name": "OPENWEATHER_API_KEY" }],
  "configFields": [
    {
      "key": "units",
      "label": "Units",
      "type": "enum",
      "description": "Temperature units",
      "options": [
        { "value": "metric", "label": "Metric" },
        { "value": "imperial", "label": "Imperial" }
      ]
    }
  ]
}
```

The manifest's `provides.tools` entry controls how your tool is loaded:

| Field | Description |
|---|---|
| `entry` | Relative path to the entry module |
| `sandbox` | `"compartment"` (default) or `"host"` |
| `permissions` | Capabilities to grant in compartment mode |
| `requireApproval` | Default approval requirement for all tools in this entry |

### Define the tool

Export a `Tool` object — a plain object with metadata, JSON Schema parameters, and an `execute` function:

```ts
// src/weather.ts
import type { Tool } from '@pandorakit/sdk/tools'

export const weatherTool: Tool = {
  id: 'weather',
  name: 'Weather',
  description: 'Get current weather for a city',
  parameters: {
    type: 'object',
    properties: {
      city: { type: 'string', description: 'City name' },
    },
    required: ['city'],
  },
  annotations: {
    readOnlyHint: true,
    destructiveHint: false,
    idempotentHint: true,
  },
  execute: async (input: { city: string }, { env, logger }) => {
    const key = env.OPENWEATHER_API_KEY
    logger.log(`Fetching weather for ${input.city}`)
    const res = await fetch(
      `https://api.openweathermap.org/data/2.5/weather?q=${input.city}&units=metric&appid=${key}`
    )
    return res.json()
  },
}
```

- `Tool` is a plain object — no framework dependencies at runtime. Only type-only imports from `@pandorakit/sdk` (erased by the TypeScript compiler).
- The `execute` function receives `{ env, logger }` as its second argument, giving tools access to environment variables and a structured logger.
- **`annotations`** are MCP-compatible hints that help the UI decide whether to auto-approve or prompt for confirmation.

### Export the tools array

The entry point (declared in the manifest's `provides.tools.entry`) exports a named `tools` array:

```ts
// src/index.ts
import { weatherTool } from './weather'

export const tools = [weatherTool]
```

Tools are registered with namespaced IDs: `pluginId:toolId` (e.g. `@pandorakit/weather:weather`).

### Enable it

Pandora automatically discovers your plugin via its `pandora.manifest.json`. Users control it from the **Plugins** page:

- Toggle the plugin on/off
- Configure settings from your `configFields`
- Optionally enable **Require Approval** on individual tools

Every plugin must be explicitly enabled on the Plugins page. A plugin with required `configFields` is skipped until the user fills them in.
