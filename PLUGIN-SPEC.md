# Plugin Architecture Spec

Design for the Pandora plugin system: sandboxed execution via SES Compartments, unified plugin manifest, plugin store, and distribution model.

---

## Mental Model

A Compartment is a **browser-like JavaScript environment**. It has pure ECMAScript — all the frozen intrinsics (`Array`, `Object`, `Map`, `Set`, `Promise`, `JSON`, `RegExp`, etc.) — but no Node.js APIs. No `fs`, no `Buffer`, no `process`, no `__dirname`. The plugin gets only what we explicitly provide: its declared npm dependencies (each in their own Compartment), scoped globals like `fetch` and `console`, and nothing else.

**Write your plugin like it runs in a browser.** Use `fetch` for network. Use the APIs the host gives you. Use npm packages that are pure JS. If you genuinely need native Node.js access, declare `sandbox: 'host'`.

---

## Principles

1. **Compartmentalize by default.** All plugin code — first-party and third-party — runs inside an SES Compartment. No distinction between "ours" and "theirs."
2. **POLA (Principle of Least Authority).** Plugins start with zero authority. They receive only the modules they declare as dependencies and the permissions they declare in their manifest.
3. **Agent-written code is always sandboxed.** No opt-out. Generated tools run in Compartments with scoped endowments only — no module imports.
4. **Host escape hatch exists but is visible.** Plugins that genuinely need host access (native addons, process spawning) declare `sandbox: 'host'`. The UI shows this prominently.
5. **No Node.js builtins.** Compartments provide zero Node.js built-in modules by default. No `fs`, `net`, `crypto`, `buffer`, `path`, `stream`, or any other Node.js API. This is the SES/Endo stance — all host capabilities are denied unless explicitly granted. Plugins that need builtins must use `sandbox: 'host'`.

---

## Execution Tiers

### Tier 1: Plugin Packages (default)

All tool, agent, and channel plugins — whether first-party (`@pandorakit/tavily-search`) or third-party (`some-community-plugin`) — run inside a Compartment with controlled module access.

**What the plugin gets:**
- Its declared `dependencies` from `package.json`, each in their own Compartment
- Its declared `peerDependencies` (e.g., `zod`), provided by the host
- Scoped endowments based on declared `permissions` (fetch, env, fs, time, random)

**What the plugin cannot do:**
- Import modules not in its `package.json` dependencies
- Access `process`, `child_process`, `fs`, `net`, or any Node.js built-in directly
- Read environment variables not declared in `permissions.env`
- Make network requests to hosts not declared in `permissions.network`
- Modify shared intrinsics (frozen by `lockdown()`)

**Important nuance:** Each dependency also runs in its own Compartment (the compartment mapper creates one per package). A dependency like `@tavily/ai-sdk` gets its own Compartment with its own declared dependencies linked in. The entire dependency tree is compartmentalized — not just the plugin's own code.

### Tier 2: Agent-Written Code (strictest)

Code generated by an LLM or written by a user through the tool generation UI. Always a code string, never a module.

**What it gets:**
- Scoped endowments only, based on declared permissions:
  - `fetch` — restricted to declared hostnames, SSRF-protected
  - `env.get()` — restricted to declared key names
  - `readFile()` — restricted to declared directory prefixes
  - `Date`, `Intl` — if `permissions.time` is true
  - `Math.random()` — if `permissions.random` is true
  - `console` — tamed (prefixed with `[sandbox]`)
- `compartment.evaluate()` — no module system, no imports
- JSON serialization boundary on output (strips prototypes, 1MB limit)

**What it cannot do:**
- Import anything — no `import`, no `require`, no dynamic `import()`
- Access any module namespace
- Escape the Compartment in any way

**No opt-out.** Agent-written code cannot declare `sandbox: 'host'`.

### Tier 3: Host Mode (escape hatch)

For plugins that genuinely cannot run in a Compartment.

**When this is needed:**
- Native addons (sharp, better-sqlite3) — C++ bindings can't load in Compartments
- Process spawning — tools that exec child processes
- Direct Node.js API access — tools that need `net`, `dgram`, `worker_threads`, etc.

**Requirements:**
- Plugin declares `sandbox: 'host'` explicitly on each tool that needs it
- The UI shows a prominent warning: "This tool runs with full host access"
- User must explicitly enable the plugin (not auto-enabled)
- First-party plugins may use host mode where necessary, but should minimize it

## Import Resolution

The central question: how does the system know which modules to attach to a plugin's Compartment?

### Answer: `package.json` is the declaration

Every npm package already declares its dependencies. This is the natural, existing, universal declaration of what modules a plugin needs. No new declaration format needed.

```
package.json
├── dependencies          → each gets its own Compartment, linked to the plugin
├── optionalDependencies  → same, but graceful failure if not installed
├── peerDependencies      → provided by the host (zod, etc.)
└── devDependencies       → NOT included (build/test only, e.g. @pandorakit/core for types)
```

### How it works (with Endo tooling)

The `@endo/compartment-mapper` already solves this. It:

1. Reads `package.json` files to build a **compartment map** — a graph of every package needed
2. Creates **one Compartment per package** in the dependency tree
3. Links module namespaces between Compartments (package A's import of package B resolves to B's Compartment exports)
4. Handles CJS, ESM, and JSON modules
5. Supports `exports` and `main` fields, subpath exports, conditions
6. Supports **policy** for per-package authority (globals, built-in modules)

### Example: `@pandorakit/tavily-search`

```json
{
  "dependencies": { "ai": "^6.0.0", "zod": "^3.0.0" },
  "optionalDependencies": { "@tavily/ai-sdk": "*", "@exalabs/ai-sdk": "*" },
  "devDependencies": { "@pandorakit/core": "workspace:*" }
}
```

The compartment mapper creates:
- A Compartment for `@pandorakit/tavily-search` — the plugin's own code
- A Compartment for `ai` — the AI SDK
- A Compartment for `zod` — schema validation
- A Compartment for `@tavily/ai-sdk` (if installed)
- A Compartment for `@exalabs/ai-sdk` (if installed)
- Module links between them based on declared dependencies

`import { tool } from 'ai'` inside the plugin resolves to the `ai` Compartment's exports. `import fs from 'fs'` throws — `fs` is not a declared dependency.

Note: `@pandorakit/core` is a `devDependency` — used only for TypeScript types during development. Plugins don't import anything from core at runtime; they simply export the right shapes (plain objects and functions) that core consumes.

### Dynamic imports

Plugins can use `await import()` for optional backends. Inside a Compartment, dynamic imports go through the Compartment's `importHook`. The compartment mapper handles this — optional dependencies are resolved if present, missing ones throw as expected.

### Node.js built-ins as "exit modules"

The compartment mapper treats Node.js built-ins as **exit modules** — they must be explicitly provided by the host. If a plugin's dependency tree includes a package that uses `fs` or `net` internally, the host must grant that exit module through policy. By default, no exit modules are provided.

This is a key difference from running in Node.js directly: packages that use Node.js built-ins won't work unless the host explicitly grants them.

## Endo Toolchain

The Endo project (by Agoric, used in production by MetaMask) provides the full toolchain. These are not experimental — they're audited and production-proven.

### `@endo/compartment-mapper` — The core

Reads `package.json` files, builds compartment maps, creates linked Compartments. This is the primary tool for loading plugin packages.

Two runtime modes:

**`importLocation`** — Load and execute from filesystem:
```typescript
import { importLocation } from '@endo/compartment-mapper'

const { namespace } = await importLocation(
  readPowers,
  pluginEntryUrl,
  {
    globals: { console, fetch },  // what the root compartment sees
    policy: { ... },              // per-package authority (optional)
  },
)
```

**`loadLocation`** — Load without executing (defer or execute multiple times):
```typescript
const application = await loadLocation(readPowers, pluginEntryUrl, options)

// Later, with specific globals:
const { namespace } = await application.import({
  globals: scopedEndowments,
})
```

### `@endo/bundle-source` — Build-time bundling

Bundles a plugin and its dependency tree into a self-contained archive. The default `endoZipBase64` format includes:
- Pre-compiled module sources for every package
- A `compartment-map.json` manifest
- SHA-512 integrity hashes

```bash
# At build/install time:
bundle-source --cache-json bundles plugin-entry.js plugin-name
```

### `@endo/import-bundle` — Runtime bundle loading

Loads a bundle into a Compartment at runtime. The bundle is isolated — only gets globals and modules you provide.

```typescript
import { importBundle } from '@endo/import-bundle'

const namespace = await importBundle(bundle, {
  endowments: { console, fetch: scopedFetch },
  // ... other compartment options
})
```

### `@endo/check-bundle` — Integrity verification

Verifies internal and external hashes of a bundle before loading. Ensures the bundle hasn't been tampered with.

```typescript
import { checkBundle } from '@endo/check-bundle'
await checkBundle(bundle) // throws if integrity check fails
```

### `@endo/module-source` — Low-level module compilation

Compiles ES module source into the record format SES Compartments expect. Used internally by the compartment mapper. Useful for loading individual module files (e.g., for agent-written code that's stored as module source).

## Two Approaches

### Approach A: Runtime loading with `@endo/compartment-mapper`

Use `importLocation` / `loadLocation` at runtime to load plugins directly from the filesystem (i.e., from `node_modules`).

**Flow:**
1. Plugin is installed via npm/bun as a regular dependency
2. At startup, `loadLocation` reads the plugin's source from `node_modules`
3. Compartment mapper builds the compartment map from `package.json` files
4. Plugin code executes in Compartments
5. Plugin exports (ToolPlugin, AgentPlugin, etc.) are extracted and registered

**Pros:**
- No build step for plugins — install and run
- Compartment mapper handles all the complexity (CJS/ESM, exports maps, etc.)
- Policy support for fine-grained per-package authority

**Cons:**
- Filesystem scanning at startup (can be cached)
- Needs `@endo/module-source` at runtime (heavier)

### Approach B: Build-time bundling with `@endo/bundle-source` + `@endo/import-bundle`

Bundle plugins at build/install time. Load bundles at runtime.

**Flow:**
1. Plugin is installed via npm/bun
2. A postinstall step runs `bundle-source` on each plugin → produces bundle JSON
3. At startup, `importBundle` loads each bundle into a Compartment
4. Plugin exports are extracted and registered

**Pros:**
- Faster startup (no filesystem scanning, no runtime parsing)
- Bundle integrity verification with SHA-512 hashes
- Bundles are portable/cacheable

**Cons:**
- Requires a build step (postinstall hook or explicit command)
- Plugin authors don't need to do anything different, but the build pipeline is more complex

### Decision: Approach A now, Approach B for the plugin store

**Start with Approach A** (`importLocation`). It's simpler, has no build step, and the compartment mapper handles everything.

**Approach B is the future distribution format for the plugin store.** The `endoZipBase64` bundle is essentially an `.xpi`/`.crx`/`.vsix` equivalent — a single self-contained archive with pre-compiled modules, a compartment map manifest, and SHA-512 integrity hashes. This is a much better fit for store distribution than raw npm packages:

| Concern | Approach A (npm install) | Approach B (precompiled bundle) |
|---------|--------------------------|-------------------------------|
| Distribution | Full npm dependency tree | Single file |
| Integrity | npm checksums only | SHA-512 per-module hashes |
| Supply chain | Transitive deps can update | Bundle is frozen at build time |
| Hot reload | Needs filesystem writes | Load from memory, no filesystem |
| Startup | Filesystem scan + parse | Decode + load |
| Store review | Review source, deps can change | Review exactly what ships |

The envisioned plugin store flow:

1. Plugin author develops locally using Approach A (install + `importLocation`)
2. Author runs `pandora build` (wraps `bundle-source`) → produces a `.pandora` bundle
3. Bundle is published to npm (as a package artifact) and indexed by the Pandora store
4. User browses store in the UI, clicks "Install"
5. Server downloads the bundle file, verifies integrity with `checkBundle`, loads with `importBundle`
6. No `npm install`, no `node_modules`, no filesystem scanning

This transition is additive — Approach B builds on A, not replaces it. Local development and first-party plugins can keep using Approach A indefinitely.

## What Changes

### `plugins.ts` — Plugin Registration

Currently plugins are statically imported in the host:

```typescript
// Current: runs entirely in host
import websearch from '@pandorakit/tavily-search'
registerToolPlugin(websearch)
```

Changes to load plugins into Compartments:

```typescript
// New: plugin code runs in Compartment
const websearch = await loadPlugin('@pandorakit/tavily-search', {
  globals: scopedEndowments,
})
registerToolPlugin(websearch.default)
```

### `ToolExport` — The universal tool interface

All plugins (host and compartment mode) export `ToolExport` objects — plain objects with JSON Schema parameters, no framework dependencies. The core wraps these into Mastra tools at load time via `bindToolExport()`. When a tool runs inside a Compartment, its `execute` closure only has access to what the Compartment provides.

### `resolveTools()` hook — Returns `ToolExport[]`

The `resolveTools` hook replaces the old `getTools` hook. It returns `ToolExport` objects instead of raw `ToolRecord`, meaning resolved tools go through `bindToolExport()` + `registerManifest()` and get full manifest support. Dynamic imports go through the Compartment's `importHook`, which the compartment mapper controls.

### Tool Permission Enforcement

Currently `ToolPermissions` are declared but not enforced at runtime. With Compartment execution, the Compartment's scope naturally enforces them — no `fetch`, `Date`, `Math.random()`, etc. unless endowed.

For plugin tools, permissions can also be enforced through the compartment mapper's **policy** system, which allows per-package authority grants.

### Agent-written code — Wire existing infrastructure

`executeInCompartment()` in `packages/core/src/tools/sandbox/compartment.ts` already handles this correctly. It just needs to be called from the tool execution path when a tool's `sandbox` mode is `'compartment'` and it's not a plugin (i.e., it's generated code).

## Policy

### How dependency permissions work

The plugin's `ToolPermissions` propagate to its **entire dependency tree**. Every Compartment in the tree gets the same scoped endowments. There is no per-dependency policy — a plugin is one trust unit.

If `@pandorakit/tavily-search` declares `permissions.network: ['api.tavily.com']`, then `@tavily/ai-sdk` (its dependency) also gets `fetch` scoped to `api.tavily.com`. There's no reason a dependency should reach hosts the plugin didn't declare.

```typescript
// The plugin declares:
permissions: {
  network: ['api.tavily.com', 'api.brave.com'],
  env: ['TAVILY_API_KEY', 'BRAVE_API_KEY'],
  time: true,
}

// Every Compartment in the plugin's dependency tree receives:
// - fetch scoped to api.tavily.com, api.brave.com (SSRF-protected)
// - env.get() scoped to TAVILY_API_KEY, BRAVE_API_KEY
// - Date, Intl
// - console (tamed)
// - NO Node.js builtins (fs, net, child_process, etc.)
// - NO process, NO unscoped fetch, NO Math.random (unless declared)
```

This means:
- The plugin author declares everything the tool needs — for itself and its dependencies
- The user sees one permission set per plugin, not per package
- No need for a separate policy file or per-dependency configuration

### Node.js builtins

Blocked for all Compartments by default. No `fs`, `net`, `http`, `child_process`, `crypto`, `buffer`, `stream`, `path`, `url`, `worker_threads`, etc.

If a dependency deep in the tree needs a builtin (e.g., a package that uses `crypto` for hashing), it won't work. The plugin must either:
- Use `sandbox: 'host'` (escape hatch), or
- Use a dependency that doesn't require builtins (e.g., a pure JS crypto library)

This is a real constraint. It will break some dependency chains. But it's the right default — builtins are the primary vector for file access, network access, and process control.

### Future: per-dependency policy

If the flat propagation model proves too coarse, per-dependency policy can be added later using the compartment mapper's policy system or LavaMoat's TOFU (Trust On First Use) approach — auto-generate a policy by observing what each package accesses, then lock it down. MetaMask uses this exact pattern. But it's not needed for the initial implementation.

## Compatibility Considerations

### The "override mistake"

SES freezes all intrinsics. Some npm packages assign to inherited properties (e.g., `obj.toString = ...`), which fails on frozen prototypes. This is tracked at [endojs/endo#576](https://github.com/endojs/endo/issues/576).

Pandora already sets `overrideTaming: 'severe'` in `ses-lockdown.ts`, which provides maximum compatibility. Most popular packages work. Need to test: `ai`, `zod`, `grammy`, `@tavily/ai-sdk`, `@ai-sdk/*` providers.

### CJS / ESM interop

The compartment mapper handles both CJS and ESM modules, including cross-format imports (CJS importing ESM and vice versa). It uses `package.json` `type` field and file extensions (`.mjs`, `.cjs`) to determine module format — same heuristics as Node.js.

### TypeScript

The compartment mapper supports TypeScript type erasure for workspace packages (using `ts-blank-space`). For published npm packages, TypeScript is already compiled to JS in `dist/` — no issue.

### Source maps

Both `@endo/module-source` and `@endo/bundle-source` support source maps. The `sourceMapHook` option allows collecting source maps during compilation for debugging. Stack traces should remain useful even with compartmentalized code.

## Resolved Questions

1. ~~**Exit modules**~~ — **None by default.** No Node.js builtins are provided to Compartments. This is the SES/Endo stance — nobody in the ecosystem maintains a "safe subset" because the philosophy is that all host capabilities start at zero. Plugins that need builtins use `sandbox: 'host'`. See Principle 5.

2. ~~**Performance**~~ — **Approach A (runtime `importLocation`) for now.** Pandora runs as a long-lived server, so a few extra seconds at startup is irrelevant. If startup overhead becomes an issue later, move to Approach B (build-time bundling) as an optimization — it's additive, not a rewrite.

3. ~~**Plugin discovery**~~ — **Manifest file + store.** Plugins ship a `pandora.manifest.json` alongside `package.json`. The manifest carries all metadata the system and store need: plugin type, display name, description, icon, categories, permissions, minimum Pandora version, etc. Discovery works by scanning installed packages for this manifest file. Distribution is via the **public npm registry** with a **curated store layer** on top (Pandora maintains an index of reviewed/approved plugins). Users install plugins entirely through the UI — no code, no config files.

4. ~~**Hot reload**~~ — **Required.** The plugin store flow (browse → install → enable) must work without restarting the server. `loadLocation` returns an Application that can be imported with specific options, so loading a new plugin into a fresh Compartment at runtime is feasible. Unloading/replacing requires clearing the plugin's registration and Compartment. Needs design work.

5. ~~**Policy generation**~~ — **Handled by `package.json`.** Since no Node.js builtins are provided, the only policy is which packages can import which — and `package.json` dependencies already declare that. The compartment mapper reads these declarations directly. No separate policy file or generation step needed.

6. ~~**First-party plugin migration**~~ — **Most plugins are already Compartment-compatible.** The AI SDK (`ai`, `@ai-sdk/*`) uses `globalThis.fetch` for all HTTP calls — no Node.js builtins. Our tool plugins (`tools-websearch`, `tools-datetime`) and agent plugins (`agent-research`) use the AI SDK + `zod` + `fetch`, all pure JS. They should work in Compartments with just scoped `fetch` and `env` endowments. Channel plugins (`channel-telegram`) need `sandbox: 'host'` because `grammy` requires WebSocket/HTTP Node.js builtins.

7. ~~**Endo version pinning**~~ — **Pin when implementing.** Lock Endo packages to a tested version. Agoric and MetaMask provide stability signals.

8. ~~**Store curation**~~ — **API-based.** The store is an API that indexes npm packages with valid `pandora.manifest.json` files. Users can point Pandora at any store API (including their own). The default store is maintained by us with review/approval. Store API returns plugin metadata, trust level (first-party / reviewed / community), and download URLs.

## Plugin Manifest

### Unified plugin model

Currently Pandora has five separate plugin types (`ToolPlugin`, `AgentPlugin`, `ChannelPlugin`, `StoragePlugin`, `VectorPlugin`) each with their own registration function. They all share the same base fields (`id`, `name`, `schemaVersion`, `envVars`, `configFields`) but differ in what they provide:

| Plugin Type | Provides |
|---|---|
| Tool | `tools[]`, `resolveTools()` |
| Agent | `agents[]` |
| Channel | `factory()` → `ChannelAdapter` |
| Storage | `factory()` → `StorageResult` |
| Vector | `factory()` → `VectorResult` |

The manifest unifies these. **A single package can provide any combination** — tools + agents, a channel + tools, storage + vector, etc. The manifest declares what the package provides and the system routes each part to the correct registry.

### `pandora.manifest.json`

```json
{
  "$schema": "https://pandora.dev/schemas/manifest-v1.json",
  "manifestVersion": 1,

  "id": "pandora-plugin-weather",
  "name": "Weather Tools",
  "description": "Current weather and forecasts for any city",
  "author": "Weather Co",
  "icon": "https://example.com/weather-icon.png",
  "version": "1.0.0",
  "homepage": "https://github.com/example/pandora-plugin-weather",
  "repository": "https://github.com/example/pandora-plugin-weather",
  "license": "MIT",

  "pandora": ">=0.0.1",

  "provides": {
    "tools": {
      "entry": "./dist/tools.js",
      "sandbox": "compartment",
      "permissions": {
        "network": ["api.weatherapi.com"],
        "env": ["WEATHER_API_KEY"],
        "time": true
      }
    },
    "agents": {
      "entry": "./dist/agents.js",
      "sandbox": "compartment",
      "permissions": {
        "network": ["api.weatherapi.com"],
        "env": ["WEATHER_API_KEY"],
        "time": true
      }
    }
  },

  "envVars": [
    { "name": "WEATHER_API_KEY", "required": true }
  ],

  "configFields": [
    {
      "key": "units",
      "label": "Temperature Units",
      "type": "enum",
      "options": [
        { "value": "celsius", "label": "Celsius" },
        { "value": "fahrenheit", "label": "Fahrenheit" }
      ]
    }
  ]
}
```

Each key in `provides` declares a capability with its own entry point, sandbox mode, and permissions. The system loads each entry point independently — tools into a Compartment, channels into host mode, etc. No guessing at export names; the manifest says exactly which file to load for each capability.

### Combined plugin example

A Telegram package that provides a channel adapter (needs host for WebSocket) and tools (pure JS, sandboxed):

```json
{
  "id": "pandora-telegram",
  "name": "Telegram",
  "description": "Chat with your assistant through Telegram",
  "author": "Pandora",
  "version": "0.0.1",
  "repository": "https://github.com/example/pandora",
  "license": "MIT",
  "pandora": ">=0.0.1",
  "manifestVersion": 1,

  "provides": {
    "channels": {
      "entry": "./dist/channel.js",
      "sandbox": "host"
    },
    "tools": {
      "entry": "./dist/tools.js",
      "sandbox": "compartment",
      "permissions": {
        "network": ["api.telegram.org"],
        "env": ["TELEGRAM_BOT_TOKEN"]
      }
    }
  },

  "envVars": [
    { "name": "TELEGRAM_BOT_TOKEN", "required": true }
  ]
}
```

The channel entry loads in host mode (grammy needs Node.js builtins). The tools entry loads in a Compartment with scoped `fetch` to `api.telegram.org`. Two separate entry points, two separate isolation contexts, one package.

### Field reference

**Identity & compatibility:**

| Field | Type | Required | Description |
|---|---|---|---|
| `manifestVersion` | `number` | Yes | Manifest schema version. Currently `1`. |
| `id` | `string` | Yes | Unique plugin identifier. Must match `package.json` `name`. |
| `name` | `string` | Yes | Human-readable display name. |
| `description` | `string` | No | Short description (1-2 sentences). |
| `pandora` | `string` | Yes | Semver range of compatible Pandora versions. |

**Metadata (all optional, top-level):**

| Field | Type | Required | Description |
|---|---|---|---|
| `author` | `string` | No | Plugin author name. |
| `icon` | `string` | No | URL to an icon image. Displayed in the UI as 32px on cards, 48px in dialogs. Falls back to a colored letter-initial. |
| `version` | `string` | No | Plugin version string (e.g., `"1.0.0"`). |
| `homepage` | `string` | No | Project homepage URL. |
| `repository` | `string` | No | Source code repository URL. |
| `license` | `string` | No | License identifier (e.g., `"MIT"`, `"Apache-2.0"`). |

These fields were originally nested under a `store` sub-object but have been promoted to the manifest top-level. `categories` and `screenshots` were dropped — they will be re-added when the plugin store is built.

**Provides (per-capability):**

Each key in `provides` (`tools`, `agents`, `channels`, `storage`, `vector`) is an entry object **or an array of entry objects**:

| Field | Type | Required | Description |
|---|---|---|---|
| `entry` | `string` | Yes | Relative path to the entry point file for this capability. |
| `sandbox` | `'compartment' \| 'host'` | No | Execution mode for this entry. Default `'compartment'`. |
| `permissions` | `ToolPermissions` | No | Required when `sandbox: 'compartment'`. Declares what this entry needs. |

**Configuration (plugin-wide):**

| Field | Type | Required | Description |
|---|---|---|---|
| `envVars` | `EnvVarDescriptor[]` | No | Environment variables the plugin depends on. |
| `configFields` | `ConfigFieldDescriptor[]` | No | User-configurable fields rendered in the UI. |

### Multiple entry points per capability

Each `provides` key can be a single object or an **array of objects** — each with its own entry point, sandbox mode, and permissions. Each entry point loads into its own Compartment, so tools with different permission needs stay isolated from each other within the same plugin.

```json
"provides": {
  "tools": [
    {
      "entry": "./dist/search.js",
      "sandbox": "compartment",
      "permissions": { "network": ["api.tavily.com"], "env": ["TAVILY_API_KEY"] }
    },
    {
      "entry": "./dist/datetime.js",
      "sandbox": "compartment",
      "permissions": { "time": true }
    }
  ]
}
```

The search tools get `fetch` scoped to `api.tavily.com`. The datetime tools get `Date` access. Neither can reach the other's resources. Two Compartments, two permission scopes, one plugin package.

### Export contracts

Each capability type has a defined set of exports the system looks for in its entry point:

**Tools** (`provides.tools.entry`):
```typescript
// Both host and compartment mode: plain objects with JSON Schema parameters
export const tools: ToolExport[]
// Optional dynamic tool resolution — returns ToolExport objects (not raw ToolRecord)
export function resolveTools(ctx: ResolveToolsContext): Promise<ResolveToolsResult>
```

**Agents** (`provides.agents.entry`):
```typescript
// Each entry point exports a plain agent definition object
export const agent: AgentDefinition  // { id, name, description, instructions, tools? }
```

Agent tool dependencies (`useTools`, `modelTools`) are declared in the manifest `provides.agents` entries, not in agent code.

**Channels** (`provides.channels.entry`):
```typescript
export function factory(env, config): ChannelAdapter | null
```

**Storage** (`provides.storage.entry`):
```typescript
export function factory(env, bindings?): Promise<StorageResult>
```

**Vector** (`provides.vector.entry`):
```typescript
export function factory(env, bindings?): Promise<VectorResult>
```

### Entry point convention

Workspace plugins during development use TypeScript entry points directly (e.g., `"./src/index.ts"`) since Bun handles TypeScript natively. Published npm plugins use compiled JavaScript (e.g., `"./dist/index.js"`). The compartment-mapper also supports TypeScript type erasure for workspace packages via `ts-blank-space`.

### Implementation status

The initial manifest system uses host-mode `import()` for all plugins while compartment-mapper integration is built incrementally. The manifest still declares the intended `sandbox` mode — once compartment loading is wired up, plugins will automatically run in their declared mode without manifest changes.

### What goes where

The split is: **manifest = what you are, code = what you do.** The system reads the manifest without executing any plugin code. It knows permissions, sandbox mode, config fields, and store metadata before loading a single line of plugin JS.
